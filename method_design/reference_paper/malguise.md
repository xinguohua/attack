## 摘要

考虑到现有的基于学习的恶意软件检测在学界与工业界都取得了显著成绩，本文提出 **MalGuise**——一个实用的黑盒对抗攻击框架，用于在黑盒设置下评估现有基于学习的 Windows 恶意软件检测系统的安全风险。MalGuise 首先采用一种新颖的、**保语义**的变换——`call-based redividing`（基于 call 的再切分）——同时对恶意软件控制流图（CFG）的**节点与边**进行操纵，使得变化不易被察觉。然后，MalGuise 借助一种**基于蒙特卡洛树搜索（MCTS）的优化**，搜索一条可应用于输入 Windows 恶意软件的、最优的 `call-based redividing` 变换序列以实现规避。最后，它在遵守 Windows 可执行文件格式约束的前提下，基于该优化后的变换序列重构对抗性恶意软件文件，从而保持与原文件相同的语义。我们在三种 SOTA 基于学习的 Windows 恶意软件检测系统上系统性地评估了 MalGuise。评测结果显示，MalGuise 取得了**极高的攻击成功率**，**大多超过 95%**，并且**超过 91%** 的对抗性恶意软件保留了原始语义。此外，MalGuise 对五种商用反病毒产品的攻击成功率最高可达 **74.97%**，凸显了对真实用户而言潜在且切实的安全隐患。

## 1. 引言

随着计算机技术的持续发展，**恶意软件**（malware = malicious software 的缩写）的扩散已成为一种严重的安全威胁，能够在计算机系统上执行恶意活动。特别是，Microsoft Windows 操作系统家族（即 Windows）的广泛采用使其成为恶意软件攻击的首要目标。根据 AV-TEST [8] 的数据，2022 年前三季度新发现的 Windows 恶意软件实例约 5958 万个，约占同期所有新发现恶意样本的 95% 以上 [7]。为了对抗持续增长的 Windows 恶意软件，学界与工业界投入了大量精力，开发出基于前沿技术的 Windows 恶意软件检测方法 [12, 40, 61]。从本质上看，Windows 恶意软件检测的历史可追溯到 1990 年代的基于签名的检测——它主要依靠一个不断更新的、已收集恶意软件签名数据库，对疑似软件进行黑名单匹配。显然，基于签名的恶意软件检测无法识别**新出现的或此前未知的**恶意软件。在过去二十年里，研究界探索并采用了大量机器学习（ML）和深度学习（DL）模型用于 Windows 恶意软件检测，在本文中统称为**基于学习的 Windows 恶意软件检测** [40, 61]。借助 ML/DL 模型强大的学习能力，基于学习的 Windows 恶意软件检测已经证明了它对**新涌现的、甚至零日恶意软件**的识别能力，并因此成为当下竞争激烈的反病毒产品市场中的关键组成部分 [28, 43]。

众所周知，基于签名的恶意软件检测可以被传统的混淆技术轻易绕过，比如压缩、加密、寄存器重分配、代码虚拟化等等。然而，随着反混淆工具的广泛可用，以及基于学习的恶意软件检测的高级能力，传统混淆在规避基于学习的恶意软件检测时**基本失效** [2, 30, 48, 63]。另一方面，近年来的研究 [21, 34, 35, 37, 58] 揭示——ML/DL 模型本身在各种领域（如计算机视觉、自然语言处理）都天然地易受**对抗攻击**的影响：攻击者恶意地构造对抗样本作为目标 ML/DL 模型的输入，致使其犯错。因此，除了传统混淆之外，本文还从**对抗攻击**的视角出发，专注于规避当下的基于学习的 Windows 恶意软件检测系统。需要指出，本文**并非要替代**传统混淆，而是要**补足**它们在规避**先进的**基于学习的恶意软件检测上的局限——后者由于其在学界与工业界的广泛采用与日益突出的地位 [28, 43, 61] 而变得至关重要。

本文尤其试图探索一种在**真实黑盒设置**下的对抗攻击，目的是**有效且高效地**生成能够规避基于学习的 Windows 恶意软件检测系统的**实用对抗性恶意软件文件**。为此，我们识别出两大关键挑战（**C#1 与 C#2**），下文逐一展开。

- **C#1**：*如何在保持原始语义的同时，生成不易被防御方察觉的实用对抗性恶意软件文件？* 先前的对抗攻击主要包含：① **添加无关 API 调用** [3, 13, 26, 54]、② **部分或全局地操纵恶意软件的原始字节** [5, 19, 31, 33, 41, 51]、以及 ③ **通过注入语义 nop 来操纵恶意软件的控制流图（CFG）** [63]。我们认为，前两类对抗攻击要么**不切实际**（因为它们生成的是对抗特征而非对抗文件），要么**严格受限**于特定恶意软件检测，如 MalConv [48]。而第三类虽然对基于 CFG 的恶意软件检测显示出更好的可扩展性，但它**仅考虑粗粒度的、对 CFG 节点的变换**，使得其变化**很明显**、易被防御方发现。因此，为应对 **C#1**，我们提出一种针对 CFG 的**新颖、细粒度变换**——`call-based redividing`——它不仅操纵节点（即指令块），还**操纵边**（即两个基本块之间的控制流关系）。
- **C#2**：*如何在黑盒设置下的庞大、离散的恶意软件空间中高效搜索，使得最优的对抗性恶意软件文件能够规避基于学习的 Windows 恶意软件检测？* 我们调研了现有 SOTA 黑盒对抗攻击，包括使用代理模型的梯度估计 [3, 31, 54]、进化算法 [13, 41]、以及强化学习 [5, 63]。可以明显看出，基于梯度估计的攻击在目标系统上**严重依赖先验信息**（例如训练数据、模型架构等）；同样，基于进化算法与强化学习的攻击由于恶意软件空间庞大且离散，**计算上代价昂贵**。因此，为应对 **C#2**，我们采用一种**基于蒙特卡洛树搜索（MCTS）的优化**来高效搜索能成功规避目标恶意软件检测系统的最优对抗性恶意软件。

综上，本文提出一个实用的、面向基于学习的 Windows 恶意软件检测系统、在黑盒设置下工作的对抗攻击框架——**MalGuise**。如图 2 所示，MalGuise 首先将输入的 Windows 恶意软件表示为 CFG，并引入一种新颖的保语义变换 `call-based redividing`，它能够操纵 CFG 的**节点与边**，使变化相对于以往攻击不易察觉。然后，我们采用一种 MCTS 优化——它能够高效有效地引导 MalGuise，在黑盒设置下的庞大、离散的恶意软件空间内搜索一条最优的 `call-based redividing` 变换序列。最后，基于优化后的变换序列，MalGuise 在遵守 Windows 可执行文件约束的前提下重构对抗性恶意软件文件，从而**成功规避**目标 Windows 恶意软件检测系统，**同时保持与原始恶意软件相同的语义**。

我们将 MalGuise 与两种 SOTA 对抗攻击对比，针对三种代表性的基于学习的 Windows 恶意软件检测系统（即 MalGraph、Magic、MalConv），在一个**大规模真实数据集**——包含数十万 Windows 恶意软件与正常软件样本——上评估其攻击有效性。评测结果显示，MalGuise **对目标基于学习的 Windows 恶意软件检测系统几乎无关**（agnostic），并**一致地取得超过 95% 的攻击成功率**。同时，我们还实证验证了 MalGuise 生成的对抗性恶意软件文件以**超过 91% 的概率**保留原始语义——相比先前对抗攻击中 < 50% 的概率或者根本失败的结果，是一项显著改进。另外，为了理解真实环境中反病毒产品的安全风险，我们评估并观察到 MalGuise 对五种常用商用反病毒产品的攻击成功率可达 **11.29%–74.97%** 的范围。

综上所述，我们的关键贡献如下：
- 为理解并评估当下基于学习的 Windows 恶意软件检测的安全风险，我们提出了一个实用的黑盒对抗攻击框架 **MalGuise**，能够生成现实的对抗性恶意软件文件。
- 据我们所知，MalGuise 是**第一个**对 Windows 恶意软件的 CFG 应用**更细粒度变换**（即 `call-based redividing`）的工作——它不仅操纵节点（指令块），还操纵边（控制流关系）。
- 评测结果显示，MalGuise 不仅以**超过 95%** 的攻击成功率有效规避了 SOTA 基于学习的 Windows 恶意软件检测，还规避了五种商用反病毒产品，攻击成功率为 **11.29% 到 74.97%**。

## 2. 预备知识与威胁模型

### 2.1. 基于学习的恶意软件检测预备知识

图 1 给出了基于学习的恶意软件检测的概览。

首先，由于 ML/DL 模型只能在数值数据上工作，训练与测试样本在输入前都要经过**特征工程**预处理。形式上，特征工程可表述为 φ : 𝒵 → 𝒳——给定一个可执行文件 z（位于**问题空间** 𝒵，即 z ∈ 𝒵），它产生一个特征向量 x（位于**特征空间** 𝒳，即 x ∈ 𝒳）。然后，使用训练样本，把 ML/DL 模型采用并训练为基于学习的恶意软件检测器 f : 𝒵 → 𝒴。也就是说，给定一个可执行文件 z ∈ 𝒵，f 能预测对应标签 y ∈ 𝒴（即 y ∈ 𝒴 = {0, 1}），使得 f(z) = y，其中 y = 0 表示正常软件，y = 1 表示恶意软件。为方便记号（参见 [36, 40]），我们将基于学习的恶意软件检测记作 g : 𝒵 → ℝ——g(z) 表示 z 的**预测恶意概率**，相反的良性概率自然推断为 1 − g(z)。

> **图 1：基于学习的恶意软件检测概览。** 训练样本（带标签）→ 特征工程 x = φ(z) → 模型训练 → 训练好的模型；测试样本 → 特征工程 → 模型预测 → 输出 y（malware = 1 / goodware = 0）。

### 2.2. 威胁模型

遵循对抗机器学习中广泛使用的威胁建模框架 [47]，我们从目标、知识、能力三方面汇报威胁模型。

**对手的目标（Adversary's Goal）。** 鉴于规避基于学习的恶意软件检测的核心目的，对手把恶意软件**错分为正常软件**是高度有利可图的——反之则不是。因此，对手的主要目标是从一个输入恶意软件 z ∈ 𝒵（即 f(z) = 1）以**最少代价**派生出一个对抗性恶意软件文件 z_adv，使得 z_adv ∈ 𝒵 不仅能**规避**目标基于学习的恶意软件检测（即 f(z_adv) = 0），还能**保留与 z 相同的语义** [40]。

**对手的知识与能力（Adversary's Knowledge and Capability）。** 我们从一个意图对基于学习的恶意软件检测系统执行经典**零知识黑盒攻击**[42, 47] 的对手出发。该术语意味着对手对目标系统**没有任何先验信息**——包括训练数据、所抽取特征集、所用学习算法及其参数、模型架构与权重等。但需澄清，零知识黑盒攻击仍然保留**目标系统的少量最小信息**——包括具体检测任务（如静态或动态分析）、表示可执行文件的特征类型（如图像、序列或图）、以及查询反馈。本文中，我们将对手的知识**限定为**：知道目标系统基于**静态分析**、专注于**像 CFG 这样的高级抽象图表示**；并知道输入 z 后的预测标签 f(z) 及其概率 g(z)，**无论是带 / 不带预测概率**。此外，为确保能生成**真实的**对抗性恶意软件文件，对手具备**在遵守 Windows 可执行文件标准规范 [45] 的前提下、操纵 Windows 可执行文件**的能力。

## 3. MalGuise 设计

### 3.1. 问题定式

针对基于学习的 Windows 恶意软件检测，本文的主要目标是从给定恶意软件文件 z ∈ 𝒵 生成一个**现实的**对抗性恶意软件文件 z_adv ∈ 𝒵，使得 z_adv 不仅被**误分类为正常软件**，还**保留与 z 相同的语义**。为此，我们探索**问题空间攻击** [47]，它在问题空间内应用**保语义变换**，把 z 逐步变换为 z_adv。具体地，如式 (1) 所示，我们从定义**第一种黑盒攻击场景（带预测概率）**入手——通过**尽可能降低** z_adv 被预测为恶意的概率 g(·)：

$$
\arg\max_{\mathbf{T}}\ g(z) - g(z_{adv}) \quad \blacktriangleright\ \text{带预测概率} \tag{1}
$$

$$
\arg\min_{\mathbf{T}}\ f(z_{adv}) \quad \blacktriangleright\ \text{不带预测概率} \tag{2}
$$

$$
\text{s.t.}\quad f(z) = 1,\quad f(z_{adv}) = 0,\quad z_{adv} = \mathbf{T}(z) \in \mathcal{Z} \tag{3}
$$

$$
\mathbf{T} = T_1 \circ T_2 \circ \cdots \circ T_n \in \mathbb{T} \tag{4}
$$

其中 T ∈ 𝕋 表示一种**原子变换**——能把一个可执行文件变换为另一个保语义的可执行文件；**T** = T₁ ∘ ⋯ ∘ Tₙ 表示一个由 n 次变换组成的**有限有序序列**，把 z 一步步变换为对抗性恶意软件，即 z_adv = **T**(z)。

类似地，式 (2) 定义了**另一种更严格的黑盒设置**——对手只能获取预测标签 f(·)，**不带预测概率**——它简单地把 z_adv 的预测标签**最小化为 0**（因为 0 表示正常软件）。

### 3.2. MalGuise 的总体框架

如图 2 所示，MalGuise 的总体框架主要由三个支柱阶段组成：① **对抗变换准备**（adversarial transformation preparation），② **MCTS 引导搜索**（MCTS-guided searching），③ **对抗恶意软件重构**（adversarial malware reconstruction）——分别在 §3.2.1、§3.2.2、§3.2.3 详细展开。

> **图 2：MalGuise 总体框架概览。**
> ① 对抗变换准备：恶意软件 → 汇编代码 → 控制流图（CFG）；可用的保语义变换为：(a) 操纵 CFG 的边——`call-based redividing`；(b) 操纵 CFG 的节点——语义 NOPs 注入。
> ② MCTS 引导搜索：选择 → 扩展 → 模拟 → 反向传播 → 输出优化的控制流图代码。
> ③ 对抗恶意软件重构：对抗汇编代码 → 对抗恶意软件。

#### 3.2.1. 对抗变换准备

为准备**保语义地**操纵给定恶意软件的对抗变换，我们最初把恶意软件样本表示为 **CFG**——其中每个节点表示一个基本块，每条边表示执行过程中两个基本块之间的控制流路径。用 CFG 表示恶意软件的主要原因有二：i) CFG **封装了执行期间内在的控制流**，包含汇编指令丰富的语义与结构信息。因此，如果我们能同时操纵 CFG 的节点与边，就能以**更细粒度的方式**改变恶意软件文件的语义与结构，从而**让变化不易被防御方察觉**；ii) CFG 已在各种软件分析任务中被广泛采用 [39]，基于 CFG 的恶意软件检测在学界与工业界**都被证明技术先进且高度有效** [23, 38, 59]。因此，**如果**我们能成功攻破这些先进的、基于 CFG 的恶意软件检测代表方法，就能**展示 MalGuise 的 SOTA 攻击有效性**。

但是，如果通过**直接操纵 CFG** 将一个可执行文件变换为另一个，极易引发各种意想不到的问题——例如寻址错误或处理错误——从而使变换后的可执行文件**根本无法正常执行**，甚至**立即崩溃**。为避免这些问题并避免被潜在防御方察觉，我们提出一种**新颖的、用于 Windows 可执行文件的保语义变换**——`call-based redividing`。它**再切分**那些包含至少一条 "call" 指令的基本块，以**同时操纵 CFG 的节点与边**。

具体来说，对一个给定可执行文件，`call-based redividing` 首先**识别并标注**所有含 "call" 指令的可用基本块。然后，假设存在一个含 "call" 指令的基本块 v，它把 "call" 作为**分割线**，尝试把原始基本块 v 再切分为**三个连续的基本块**的组合——即**前基本块** v_fore、**后基本块** v_post、以及**中基本块** v_mid。为避免新的基本块 v_mid 因为仅含 "call" 与 "jmp" 两条指令而被防御方察觉，`call-based redividing` 通过在 "call" 之前或在 "call" 与 "jmp" 之间**注入语义 nop** [14, 41] 来**丰富 v_mid 中的汇编指令**，并使用 [41] 中的**上下文无关文法**来多样化地生成语义 nop。

如图 3 所示，我们首先把最新 "LockBit 3.0"（一个活跃且著名的勒索软件团伙）中一个基本块作为代表例子展示——见图 3(a)。应用变换后，图 3(b) 展示了变换后的**三连续基本块组合**——其中两个基本块（即 v_fore 和 v_mid）的末尾各有一条新加的 "jmp" 指令，并且在 v_mid 中 "call" 与 "jmp" 之间还有一些新加的、作为语义 nop 的汇编指令。

> **图 3：`call-based redividing` 把 "LockBit 3.0" 勒索软件中的一个基本块（图 3(a)）再切分为三个连续基本块的组合（图 3(b)）。**

#### 3.2.2. MCTS 引导搜索

回顾我们在 §3.1 中定式化的对抗攻击，以及 §3.2.1 中针对给定恶意软件 z（其 CFG 表示 x = φ(z)）的 `call-based redividing` 变换，我们把 MalGuise 分解为两步：先找到一个**最优变换序列 T**，它将原始 CFG x 连续地变换为对抗 CFG x_adv（在本节 §3.2.2 详述）；再基于该 T 重构最终的对抗性恶意软件文件（即 z_adv，在 §3.2.3 详述）。

本质上，我们要求解的是一个**长度为 N 的优化变换序列 T = T₁ ∘ ⋯ ∘ Tₙ**，且每个 Tₖ = {𝕀ₖᶜᵃˡˡ, 𝕀ₖˢ⁻ⁿᵒᵖˢ} 涉及两个决策：i) **选择所有可用 "call" 指令中的一条**进行再切分，即 𝕀ₖᶜᵃˡˡ——值得注意的是，𝕀ₖᶜᵃˡˡ 可以**递归地、可重复地**被选中；ii) **决定要注入的合适语义 nop**，即 𝕀ₖˢ⁻ⁿᵒᵖˢ——它可以由所用上下文无关文法 [41] **无限生成**。简而言之，在 MalGuise 中、在黑盒设置下、在有限计算预算下，求解最优 T 要求**在一个无限且离散的空间中探索与优化**。为此，我们的关键思路是**采用一种 MCTS 引导搜索算法** [17]，主要原因有二：第一，MCTS 已被广泛且成功地用于解决长期挑战性的"计算机围棋"问题 [17, 49] 以及其它需要少量或无领域知识的困难优化问题 [10]；第二，我们在黑盒设置下优化 T 的任务**严格地局限在一个无领域先验的、无限且离散的空间中搜索**。因此，我们认为 MCTS 引导搜索算法**与本任务需求高度契合**。

> **算法 1：MCTS 引导搜索算法**
> **输入**：给定恶意软件 z 及其 CFG x、目标系统 f、最大长度 N、模拟次数 S、预算 C。
> **输出**：变换序列 T。
>
> ```
> 1  Begin
> 2     𝕀ᶜᵃˡˡ ← GetAllCalls(x);
> 3     v, T ← InitMCTSRootNode(x, 𝕀ᶜᵃˡˡ), ∅;   // 初始化
> 4     for i ← 1 to N do                          // 循环到最大长度
> 5         for j ← 1 to C do                      // 循环到计算预算
> 6             if random(0, 1) < 0.5 then         // 避免无限扩展
> 7                 v_selected ← Selection(v);
> 8             else
> 9                 v_selected ← Expansion(v);
> 10            reward ← Simulation(v_selected, f, S);
> 11            BackPropagation(v_selected, reward);
> 12        v_node ← ChildWithHighestReward(v);
> 13        T ← T.append(v_node.T);
> 14        x_adv ← v_node.x;
> 15        if Evaded(f, x_adv) == True then
> 16            return T;
> 17        v ← v_node;
> ```

算法 1 给出了 MCTS 引导搜索算法——输入给定恶意软件 z 及其 CFG x，输出变换序列 T。首先，我们从 x 获取所有可用的 "call" 指令 𝕀ᶜᵃˡˡ，并初始化 MCTS 的根节点 v 与 T（第 2–3 行）。同时，我们把最优变换序列的最大长度限制为 N（第 4–17 行），把 MCTS 的最大迭代次数限制为 C（即计算预算）（第 5–11 行）。MCTS 优化过程遵循四个标准步骤——**Selection（选择）、Expansion（扩展）、Simulation（模拟）、Backpropagation（反向传播）**（第 6–11 行）。值得注意的是，由于 `call-based redividing` 可被无限执行，MCTS 的博弈树可向下被**无限扩展**——即 Expansion 阶段。因此，我们通过在已建立的博弈树上做**简单的随机采样**，**强制让选择倾向于最有希望的子节点**——即 Selection 阶段（第 6 行）。在经过 C 次迭代后，我们获取奖励最高的子节点 v_node，将其变换附加到变换序列 T，并更新对抗 CFG x_adv（第 12–14 行）。最后，如果 x_adv 成功规避 f，则返回变换序列 T；否则继续把 v_node 作为下一轮的根节点，直到达到最大长度 N。简洁起见，更多实现细节见附录 A。

#### 3.2.3. 对抗恶意软件重构

最后，我们基于原始恶意软件文件 z 与优化得到的变换序列 T，重构对抗性恶意软件 z_adv = **T**(z)——算法 2 简要给出了流程。值得注意的是，每次变换 Tₖ = {𝕀ₖᶜᵃˡˡ, 𝕀ₖˢ⁻ⁿᵒᵖˢ}；我们记 Aₖᶜᵃˡˡ 为 𝕀ₖᶜᵃˡˡ 的地址。我们先计算 T 中所有 `call-based redividing` 变换所需的空间为 Δ（第 2–6 行）。同时，分别记 S_slack 与 A_slack 为 z 的 ".text" 节中**slack 空间**的大小与地址；类似地，记 A_last 与 S_last 为**最后一节**的地址与大小。若 Δ 小于 slack 空间的大小，我们直接取 A_slack 作为**注入地址** A^inj（第 7–8 行）。否则，我们将新增一个节，其起始注入地址计算为 A^inj = A_last + RoundUp(S_last, page_size)（第 9–10 行）。这是因为根据标准 Windows 可执行文件规范 [45]，其节的大小必须是**架构页大小的整数倍**（如 x86 与 MIPS 为 4KB），以防止执行期间出现意想不到的问题（例如寻址错误）。除此之外，还须采取若干其它后续动作以满足 Windows 规范——例如设置 ".text" 节（或新增节）的大小，并调整 Windows 可执行文件头中的若干字段（例如 "size of image"）。

> **算法 2：对抗恶意软件重构**
> **输入**：原始恶意软件 z、变换序列 T。
> **输出**：对抗性恶意软件 z_adv。
>
> ```
> 1  Begin
> 2     Δ ← 0;
> 3     for k ← 1 to N do
> 4         Δ ← Δ + GetSize(𝕀ₖᶜᵃˡˡ) + GetSize(𝕀ₖˢ⁻ⁿᵒᵖˢ);
> 5         𝕀ₖʲᵐᵖ ⇔ jmp_[Aₖᶜᵃˡˡ + GetSize(𝕀ₖᶜᵃˡˡ)];
> 6         Δ ← Δ + GetSize(𝕀ₖʲᵐᵖ);
> 7     if Δ < S_slack then        // 注入 slack 空间
> 8         A^inj ← A_slack;
> 9     else                        // 注入新节
> 10        A^inj ← A_last + RoundUp(S_last, page_size);
> 11    take actions to meet the standard specifications;
> 12    z_adv ← Adv_Patch(z, T, A^inj);   // 参见算法 3
> 13    return z_adv;
> ```

获取注入地址 A^inj 后，我们用 `Adv_Patch` 过程重构 z_adv——主要见算法 3。具体来说，我们首先把所选 𝕀ₖᶜᵃˡˡ 指令**替换为一条新的 jmp 指令**（即 "jmp_[Aₖⁱⁿʲ]"），它把控制流转至**第 k 次变换的注入地址**——即 Aₖⁱⁿʲ（第 4 行）。然后，从 slack 空间或新增节的注入地址 Aₖⁱⁿʲ 开始，我们按顺序**沉积** 𝕀ₖᶜᵃˡˡ 与 𝕀ₖˢ⁻ⁿᵒᵖˢ（第 5–7 行）。随后我们注入一条新的 jmp 指令（即 𝕀ₖʲᵐᵖ ⇔ jmp_[Aₖⁱⁿʲ + GetSize(𝕀ₖˢ⁻ⁿᵒᵖˢ)]），把控制流**回转**到原节中 𝕀ₖᶜᵃˡˡ 的下一条指令（第 8–9 行）。最后，应用完全部 N 次变换后，我们重构出**保留原始恶意软件 z 相同语义**的最终对抗性恶意软件 z_adv。

> **算法 3：Adv_Patch 过程**
> **输入**：原始恶意软件 z、变换序列 T、注入地址 A^inj。
> **输出**：对抗性恶意软件 z_adv。
>
> ```
> 1  Begin
> 2     z_adv, A₁ⁱⁿʲ ← z, A^inj;
> 3     for k ← 1 to N do
> 4         z_adv ← Patch(z_adv, Aₖᶜᵃˡˡ, jmp_[Aₖⁱⁿʲ]);
> 5         z_adv ← Patch(z_adv, Aₖⁱⁿʲ, 𝕀ₖᶜᵃˡˡ);
> 6         Aₖⁱⁿʲ ← Aₖⁱⁿʲ + GetSize(𝕀ₖᶜᵃˡˡ);
> 7         z_adv ← Patch(z_adv, Aₖⁱⁿʲ, 𝕀ₖˢ⁻ⁿᵒᵖˢ);
> 8         Aₖⁱⁿʲ ← Aₖⁱⁿʲ + GetSize(𝕀ₖˢ⁻ⁿᵒᵖˢ);
> 9         z_adv ← Patch(z_adv, Aₖⁱⁿʲ, 𝕀ₖʲᵐᵖ);
> 10        Aₖ₊₁ⁱⁿʲ ← Aₖⁱⁿʲ + GetSize(𝕀ₖʲᵐᵖ);
> 11    return z_adv;
> ```

图 4 展示了对 "LockBit 3.0" 勒索软件、应用一次 `call-based redividing` 变换后所生成的对抗性 "LockBit" 的概念布局。值得注意的是，**仅对 "LockBit 3.0" 应用两次优化后的 `call-based redividing` 变换**，所生成的对抗 "LockBit" 即可**成功规避**三种 SOTA 基于学习的恶意软件检测（MalGraph、Magic、MalConv）以及我们评测中所用的著名反病毒产品 Kaspersky。

#### 4.1.3. 基线攻击

我们将 MalGuise 与两类基线攻击对比，即**对抗攻击**与**混淆**。原则上，为便于公平对比，我们对所有基线攻击都遵循其论文或实现中提供的相同评估设置。

- **对抗攻击**：我们采用两种 SOTA 对抗攻击：i) **MMO** [41] 是一种**白盒**对抗攻击，使用基于梯度的优化、配合二进制多样化工具来操纵 Windows 恶意软件的原始字节。其最大迭代次数为 200，对抗恶意软件大小的增量限制为 5%。ii) **SRL** [63] 是一种**黑盒**对抗攻击，使用强化学习迭代地把语义 nop 注入恶意软件的 CFG。其最大迭代次数、注入预算、修改基本块数分别为 30、5%、1250。值得注意的是，SRL **只生成对抗 CFG 特征而非现实恶意软件文件**，因此**与 MalConv 不兼容**——后者要求原始字节作为输入。

- **混淆**：我们采用三种**已被用于混淆 Windows 可执行文件**的混淆工具，并按其默认设置评估。特别地，**UPX** [53] 是一款通过对整个可执行文件压缩来混淆可执行文件的开源通用打包工具；**VMProtect** [56] 通常通过**代码虚拟化**进行混淆——模拟一个执行可执行文件中关键部分的虚拟机；**Enigma** [20] 采用多种混淆技术组合——例如导入表消除、API 模拟、代码虚拟化等。

#### 4.1.4. 评估指标

我们采用两类评估指标如下。

- **攻击成功率（ASR）**。ASR 是对抗攻击最常见的评估指标 [37, 41, 63]。给定候选恶意软件集合 **Z**，ASR 定义为：在所有恶意软件（即 f(z) = 1）中，**成功规避**目标系统（即 f(z_adv) = 0）的生成对抗性恶意软件的比例：

$$
ASR = \frac{|(f(z) = 1) \wedge (f(z_{adv}) = 0)|}{|f(z) = 1|},\quad \forall z \in \mathbf{Z} \tag{5}
$$

其中 |·| 计数满足条件的元素个数。

- **语义保留率（SPR）**。由于生成的对抗性恶意软件**可能并未保留与原始恶意软件相同的语义**——即不能被执行、或失去原始恶意行为——我们定义 SPR 为：在所有对抗性恶意软件中、**保留了原始语义**的比例：

$$
SPR = \frac{|Sem(z, z_{adv}) = 1|}{|(f(z) = 1) \wedge (f(z_{adv}) = 0)|},\quad \forall z \in \mathbf{Z} \tag{6}
$$

其中 Sem(z, z_adv) = 1 表示 z_adv 与 z 保留了相同语义。§4.2.3 将详细介绍如何实证地度量 SPR。

#### 4.1.5. 实现细节

MalGuise 主要用 Python 实现，并在一台配备 20 颗 Intel Xeon CPU、128 GB 内存和 4 块 NVIDIA GeForce RTX 3090 的计算机上评估。首先，MalGuise 使用 IDA Pro 6.4 [24] 中的 IDAPython 反汇编 Windows 可执行文件并把它们表示为 CFG。在 MCTS 引导搜索算法中，我们默认将最大长度 N 设为 6、计算预算 C 设为 40、模拟次数 S 设为 1，并将注入的语义 nop 大小限制为不超过原大小的 5%。为了重构对抗性恶意软件文件，我们主要采用两个 Python 库——**pefile**¹ 与 **LIEF**²——来解析与打补丁 Windows 可执行文件。值得注意的是，注入的语义 nop 大致可分为四类：算术（如 "add_eax, 1; sub_eax, 1"）、逻辑（如 "xor_eax, eax"）、比较（如 "cmp_eax, eax"）、以及数据传输（如 "push eax; pop_eax"）。

> ¹ pefile: https://github.com/erocarrera/pefile
> ² LIEF: https://lief-project.github.io

#### 4.2.1. 回答 RQ1（攻击性能）

对所有攻击而言，表 3 显示三种目标系统更低的 FPR 值对应于每次攻击取得的**更高 ASR**——原因明显：二分类器允许的更低 FPR 值意味着它有**更高的阈值**，因此对手能更容易地把预测恶意概率降到该阈值之下。此外，**显然每次攻击在 *w/o prob.* 场景下的表现都不优于 *w/ prob.* 场景**。

**对抗攻击**：MMO 在两种攻击场景下对所有三种目标系统的 ASR 都很低。特别是在 FPR = 1% 的情况下，MMO 在 *w/ prob.* 与 *w/o prob.* 两种场景下的 ASR 分别**低于 16% 与 4%**。这意味着——尽管 MMO 理论上可被用于攻击所有目标系统，但其针对这些系统的攻击性能**逊色得多**。我们推测的主要原因是：由于 MMO 通过二进制多样化技术操纵恶意软件的整个原始字节，**不**考虑不同目标系统所采用的判别性特征。

对 SRL 而言，在两种场景下、两种 FPR 下，其针对 Magic 的 ASR 都明显高于针对 MalGraph 的 ASR。这主要是因为 SRL 是**专门为攻击 Magic**（仅基于 CFG）而设计的。但 MalGraph 的**层级特性**——把函数调用图与 CFG 结合——进一步削弱了 SRL 的攻击性能，因为 SRL **只操纵 CFG 的节点而忽略对边的操纵**。

**混淆**：可以观察到，所有三种混淆的攻击性能都较差——这再次验证了"传统混淆工具对基于学习的恶意软件检测在很大程度上无效" [2, 30, 48]。具体而言，VMProtect 对所有三种目标系统的攻击性能**最差**，而 UPX 和 Enigma 攻击性能略好，但仍然不令人满意，全部 ASR 不超过 40%。这主要是因为：VMProtect 通常只混淆恶意软件文件中**一小部分** [56, 57]，而 UPX 与 Enigma 可以**全局地混淆**整个恶意软件文件，从而稍稍增加了改变判别性语义特征的可能性。然而，如果没有安全专家的介入，UPX 与 Enigma 仍然难以**有意地操纵判别性语义特征**。从大小改变比角度的更多评估见附录 C。

**MalGuise 与所有基线攻击对比**：表 3 显示，MalGuise **在所有场景和情形下**对所有三种目标系统都取得了最佳攻击性能。当攻击 MalGraph 与 Magic 时，在所有场景与情形下 MalGuise 取得的所有 ASR 值**都超过 97%**。更重要的是，即使在更严苛的 *w/o prob.* 场景下，MalGuise 的 ASR 性能**几乎不变**——相比 *w/ prob.* 场景下降不超过 1%。当攻击 MalConv 时，尽管 MalGuise 的 ASR 性能比所有基线攻击都好得多（FPR = 1% 时低于 35%），其 ASR 仍然**相对较差**。但在 §4.2.2 的后续调查中，我们发现 MalGuise 对 MalConv 的攻击性能**高度依赖于**所注入的语义 nop。因此，通过纳入最常用的 25 种语义 nop，MalGuise 得以提升——在 *w/ prob.* 与 *w/o prob.* 场景下分别对 MalConv 取得 **97.76% 与 95.18%** 的更高 ASR。

> **对 RQ1（攻击性能）的回答**：先前的混淆与对抗攻击要么无法提供令人满意的攻击有效性，要么无法很好地扩展到不同类型的基于学习的 Windows 恶意软件检测。然而，**即使**在严格黑盒攻击场景（*w/o prob.*）下，MalGuise **对基于学习的 Windows 恶意软件检测几乎无关（agnostic）**，在大多数情况下都以**超过 95%** 的攻击成功率取得高 ASR。

#### 4.2.3. 回答 RQ3（实用性性能）

按式 (6) 所定义，我们以**语义保留率（SPR）**评估 MalGuise 与两种基线对抗攻击的实用性性能——其核心是判断对抗性恶意软件 z_adv 是否与原始恶意软件 z 具有相同语义，即 Sem(z, z_adv)。由于可执行文件的**内在复杂性**，对 Sem(z, z_adv) **没有精确解** [6]，因此我们诉诸**实证验证**——通过在同一 Cuckoo 沙箱 [18] 中分别运行 z_adv 与 z、收集并对比它们调用的两条 API 序列（即 API_{z_adv} 与 API_z）来判断。如式 (7) 所示，为量化 z_adv 与 z 之间的语义差，我们计算两条 API 序列间的**归一化编辑距离** dist_norm(z, z_adv)：

$$
dist_{norm}(z, z_{adv}) = \frac{Distance(API_z, API_{z_{adv}})}{\max(l(API_z), l(API_{z_{adv}}))} \in [0, 1] \tag{7}
$$

其中 Distance(API_z, API_{z_adv}) 表示两序列之间的编辑距离，l(·) 表示序列长度。

然而，由于恶意软件在执行期间可能执行**随机动作** [27]，在同一沙箱中**两次**运行相同的恶意软件 z 所采集的 API 序列可能**不同**——这意味着 dist_norm(z, z) **几乎不可能取到 0**。因此，我们计算 dist_norm(z, z) 并将其与一个**通用距离阈值** dist_Δ 比较，以决定 Sem(z, z) ∈ {0, 1}。为确定 dist_Δ，我们首先在同一沙箱中**两次运行所有原始恶意软件样本**，然后从所有对应 dist_norm(z, z) 中选**第 99.5 百分位**的值作为 dist_Δ。然后，如式 (8) 所示，我们最终可通过对比其归一化编辑距离 dist_norm(z, z_adv) 与 dist_Δ 来确定 z_adv 与 z 是否具有相同语义，并进而按式 (6) 评估 SPR：

$$
Sem(z, z_{adv}) = \begin{cases} 1 & \text{if } dist_{norm}(z, z_{adv}) < dist_\Delta \\ 0 & \text{otherwise} \end{cases} \tag{8}
$$

由于上述评估过程**极为耗费资源与时间**，我们从所有在 §4.2.1 中成功规避对应目标系统的对抗性恶意软件中**随机选取 10%** 用于后续评估，并把评估结果汇报于表 5。显然，**SRL 的实用性性能不适用评估**——因为它只生成对抗特征而非现实的对抗性恶意软件文件。对 MMO 而言，可观察到，其对三种目标系统取得的 SPR 值都**处于低水平**——大致在 40% 到 50% 之间。这表明 MMO 生成的对抗性恶意软件中**少于 50% 保留原始语义**。然而，MalGuise 在所有三种目标系统上都取得**最佳实用性性能**——**SPR 超过 91%**——这展示了 MalGuise 在保留原始语义方面的**最佳有效性**。

**9% 失败的原因**。为调查上述评估中约 9% 的失败原因，我们用 IDA Pro [24] 与 OllyDbg [62] 进行**人工审查**，并揭示了两个主要原因：第一，少量恶意软件样本含有 **overlay**——它不是官方 Windows 可执行文件格式的一部分，但通常被用于执行恶意行为。尽管 MalGuise 遵循标准 Windows 可执行文件格式规范，对抗性恶意软件重构阶段可能**影响那些少数恶意软件样本的 overlay**，从而无法被完全或正确执行。第二，在处理 "call" 指令时把返回地址 **push/pop** 到栈上是 Windows 可执行文件的标准过程——然而少量含 **junk code** 的恶意软件可能**不遵循上述过程**（例如不弹出返回地址）。这些例外可能让 MalGuise 失效，因为它遵循标准 Windows 规范来重构对抗性恶意软件。

> **对 RQ3（实用性性能）的回答**：先前对抗攻击要么只生成**不可执行**的对抗"特征"，要么生成的对抗性恶意软件中很大一部分**丧失原始语义**。然而，MalGuise 展示了**最佳实用性性能**——**超过 91%** 的生成对抗性恶意软件文件保留了原始语义。

#### 4.2.5. 回答 RQ5（可能的防御）

| :--- | ---: | ---: | ---: | ---: |
| MalGraph | 97.47 | 82.25 | 82.43 | 80.26 (-17.21) |
| Magic | 99.29 | 97.94 | 96.91 | 93.85 (-5.44) |
| MalConv | 97.76 | 80.07 | 75.15 | 74.63 (-23.13) |

表 8 显示：无论采用死代码消除、CFG 缩简，还是二者的结合，MalGuise 取得的 ASR 性能都**有所下降**——表明被防御系统的鲁棒性性能确有提升。但即使在二进制代码优化防御下，MalGuise 仍取得**相对较高**的攻击性能——ASR **超过 74%**。

**(3) 基于启发式的对抗检测**：为评估**完全了解 MalGuise 实现细节**的防御方的影响，我们实现一种基于启发式对抗检测的**自适应防御**——它包含两条启发式规则 ℝ₁ 与 ℝ₂。特别地，ℝ₁ 指：除 ".text" 节外，存在另一个或多个**带可执行权限**且**固定地址**的节；ℝ₂ 指：存在至少一个基本块，其中含一条 "call" 指令后跟一条 "jmp" 指令、中间有语义 nop。需要注意的是，由于 MalGuise 用上下文无关文法 [41] 迭代生成语义 nop，**判定一段特定代码是否语义 nop** 是一个被广泛认知的**非平凡、不可判定**的问题 [15, 50]，因此我们采用 Optimice [25] 来检测如上语义 nop。设置上，我们首先从所有成功规避三种基于学习的 Windows 恶意软件检测系统的对抗性恶意软件中**随机选 15,000 条**，并从基准数据集中选 15,000 条正常软件，然后用上述对抗检测来检验是否能被检出，最终把检测结果展示于表 9。

**表 9：基于两条启发式规则的对抗检测的检测性能（即 TPR 与 FPR）。**

| 对抗检测 | TPR | FNR = (100% − TPR) | FPR |
| :--- | ---: | ---: | ---: |
| ℝ₁-based | 70.15 | 29.85 | 5.66 |
| ℝ₂-based | 96.91 | 3.09 | 52.16 |
| ℝ₁ & ℝ₂-based | 67.75 | 33.25 | 2.95 |

可观察到，ℝ₁-based 对抗检测的 TPR 与 FPR 都低——意味着**绝大多数正常软件确实只有 ".text" 节**。然而，也有大量对抗性恶意软件**只在剩余 slack 空间中注入了语义 nop**。此外，ℝ₂-based 对抗检测呈现**最佳 TPR 但最差 FPR**——说明虽然 ℝ₂ 可被用于检测**几乎所有对抗性恶意软件**，但**超过 50% 的正常软件**会被误分类为对抗性恶意软件。与 ℝ₁ 类似，结合 ℝ₁ 与 ℝ₂ 的对抗检测取得约 67.7% 的 TPR——意味着约 **33.2%** 的对抗性会被误分类为正常软件（即规避对应的对抗检测）。此外，它也显示 **2.9% 的 FPR**——这高于一般恶意软件检测系统所能容忍的 FPR 上限（如 1% 或 0.1%）。

综上，启发式对抗检测的自适应防御可在一定程度上检测 MalGuise 生成的对抗性恶意软件——前提是它被装备了 MalGuise 的所有实现细节。然而，它也暴露出**TPR 与 FPR 无法同时平衡**的困境——即未能提供令人满意的检测性能。

**(4) 模糊哈希（Fuzzy hashing）**：由于 MalGuise 被优化为**使用尽可能少的变换**以保留相同语义，这也使得防御方有机会使用**基于模糊哈希**的恶意软件分析（如 ssdeep [32]、TLSH [46]、SpamSum [29] 等）做防御——这已被大量研究 [11, 60]。为实现基于模糊哈希的防御，我们首先把训练集中的所有恶意软件样本提交以**初始化**恶意软件数据库，然后用验证集**确定满足 FPR = 1% 与 FPR = 0.1% 的阈值**以便公平对比。之后，在测试集中评估基于模糊哈希的防御的检测性能——见表 10。通过对比表 2 与表 10，我们发现：所有基于模糊哈希的防御都呈现**更不稳定且更差**的检测性能。

| 基于模糊哈希的防御 | AUC | FPR=1% TPR | FPR=1% bACC | FPR=1% ASR | FPR=0.1% TPR | FPR=0.1% bACC | FPR=0.1% ASR |
| :--- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| ssdeep | 83.92 | 57.61 | 78.70 | 52.94 | 10.16 | 55.31 | 100 |

我们进一步**强化**这三种基于模糊哈希的防御——把测试集中所有恶意软件样本都提交到**恶意软件数据库**，意味着所有这些防御都对原始恶意软件样本有**完整知识**、可使用模糊哈希做检测。最终，我们评估 MalGuise 对这些**被强化的防御**的 ASR 性能，见表 10 "ASR" 列。可清楚观察到——在 FPR = 1% 的情况下，MalGuise 对所有三种被强化防御的 ASR **均超过 50%**；在 FPR = 0.1% 的情况下，ASR **接近 100%**，呈现出**对基于模糊哈希的防御的高攻击成功率**。

## 5. 讨论

### 5.1. 相关工作

几乎所有对抗攻击对恶意软件检测的研究，都**主要集中于基于学习的、使用静态特征的恶意软件检测**。考虑到不同基于学习的恶意软件检测所采用特征表示的广泛多样性，研究者提出了**为这些不同检测方法量身定制的不同对抗攻击**。例如，要攻击基于 API 调用的恶意软件检测，已有一类对抗攻击通过**添加无关 API 调用**——由基于梯度的优化或贪心算法 [3, 13, 26, 54] 选择。然而，这些对抗攻击**不切实际**，因为它们生成对抗 API 调用而非现实恶意软件文件。对基于原始字节的恶意软件检测（尤其是 MalConv [48]），研究者探索了**部分修改特定区域**或**全局修改所有原始字节同时保留相同语义**。例如，[5, 19, 31, 33, 51] 都依赖在输入恶意软件的特定位置**追加或注入字节**，而 MMO [41] 用二进制多样化技术**全局操纵原始字节**。但我们认为这些对抗攻击**严格限于基于原始字节的恶意软件检测**，**无法扩展**到其它恶意软件检测。

最近，研究开始探索针对基于**更先进抽象图表示**的恶意软件检测的对抗攻击。2022 年，Zhang 等 [63] 提出了 **SRL** 用于攻击基于 CFG 的恶意软件检测——它通过强化学习引导，**顺序地把语义 nop 注入 CFG**。同样地，我们的工作也旨在规避基于 CFG 的先进恶意软件检测，但与 SRL 在三方面**根本不同**：

- **细粒度变换**。SRL 采用**粗粒度变换**，只操纵 CFG 的节点；而我们提出**更细粒度**的 `call-based redividing` 变换，**同时操纵节点与边**，使变化更不易被察觉。
- **非对抗"特征"**。如 §4.2.3 所讨论，SRL 是**特征空间**对抗攻击，本质上生成对抗"特征"；而我们的 MalGuise 能**成功生成真实的对抗性恶意软件文件**。
- **攻击泛化性**。现有对抗攻击（如 SRL）主要在**有限的基于学习的恶意软件检测模型**上评估，而 MalGuise **额外针对真实环境中的反病毒产品**评估，从而展示出更好的**攻击泛化性**。

### 5.2. 可能的使用场景

我们讨论 MalGuise 的可能使用场景如下。首先，MalGuise 作为一种实用的黑盒对抗攻击，可以**补足蓝队工具包**——解决传统混淆在规避当下基于学习的高级恶意软件检测时的局限。第二，MalGuise 可以**充当反病毒厂商研发团队的工具**——把开发中的基于学习的恶意软件检测的底层弱点暴露出来，从而**主动且有目的地**提升反病毒产品的鲁棒性。第三，全面且公正的反病毒测试对于**确保透明度**与培育用户与厂商之间的信任至关重要。鉴于对抗性恶意软件出众的规避能力，MalGuise 为像 AV-TEST [8] 这样的**第三方独立测试组织**提供了一个独特的机会，对反病毒进行综合评估。

### 5.3. 潜在伦理顾虑

本研究的首要目标是**用对抗攻击评估基于学习的 Windows 恶意软件检测的安全风险**——这是已有研究 [22, 40, 41, 47, 63, 64] 中**已被认可且有先例的话题**，主要动机是**关注潜在对手可能构造能规避检测的 Windows 恶意软件**，并通过强调**亟需更鲁棒的基于学习的 Windows 恶意软件检测**。尽管我们的意图严格限于评估安全风险，我们仍认识到与该研究关联的**潜在伦理顾虑**。因此，为在**避免潜在伦理顾虑**与**协助安全社区增强基于学习的 Windows 恶意软件检测鲁棒性**之间取得平衡，我们将代码共享**限定为仅向经过验证的学术研究者开放**——这沿用了先前研究 [22, 47, 64] 确立的先例。

## 6. 结论、局限与未来工作

本文提出一种新颖的保语义变换 `call-based redividing`——能够**同时操纵 CFG 的节点与边**——并进一步提出对抗攻击框架 **MalGuise**——在黑盒场景下针对基于学习的 Windows 恶意软件检测。广泛的评估表明：MalGuise 不仅有效规避 SOTA 基于学习的 Windows 恶意软件检测系统——**攻击成功率大多超过 95%**——还能规避五种商用反病毒产品——**攻击成功率高达 74.97%**。我们相信本研究**提升了公众对 Windows 恶意软件检测领域内对抗攻击所构成安全威胁的认识**，并**呼吁更多研究增强现有 Windows 恶意软件检测的鲁棒性**。然而，我们也认识到局限性并概述潜在未来工作如下。

- **语义保留的验证**。为了验证生成的对抗性恶意软件**保留原始语义**，我们在 §4.2.3 与式 (7) 中给出了**自动化实证验证**——通过对比两个恶意软件在同一沙箱环境下运行所调用的 API 序列。虽然实用合理，但这种实证验证**无法保证两个恶意软件在所有可能环境下严格语义等价**——因为某些恶意软件在不同环境下可能调用**随机** API。
- **基于动态分析的恶意软件检测**。MalGuise 针对的是基于学习的 Windows 恶意软件检测，它**属于静态分析**。特别地，MalGuise 的核心是操纵可执行文件的 CFG（通过 `call-based redividing` 变换）——它**不改变执行流**，从而**不影响动态分析**。因此，尽管动态分析由于过高的时间与资源消耗、部署不那么普遍，我们将针对基于动态分析的恶意软件检测生成对抗性恶意软件留作**未来工作**。
- **格式无关的对抗性恶意软件**。由于不存在普遍适用于所有不同文件格式与操作系统的恶意软件类型的分析技术，现有恶意软件分析通常会**指明目标文件格式与操作系统**。本文中，MalGuise 针对的是 PE 格式的 Windows 恶意软件，未来工作中我们将探索如何针对所有类型的恶意软件检测生成**格式无关的对抗性恶意软件**。

## 附录 A：MalGuise 中更多实现细节

算法 4 概述了 `Selection` 的主要过程。由于 MCTS 博弈树中的可能路径是**无限**的，探索博弈树中的所有节点会**显著增加计算开销**。通过采用**置信上界（Upper Confidence Bound, UCB）算法** [10]，`Selection` 可选择 MCTS 节点 v 的、**考虑访问次数与奖励值权衡**后的、**得分最高的子节点**（第 4–6 行）。

> **算法 4：Selection(v) 过程**
>
> ```
> 1  Begin
> 2     max_score ← 0;
> 3     for v_child in children of v do
> 4         exploit ← v_child.reward / v_child.visits;
> 5         explore ← √(2·ln(v.visits) / v_child.visits);
> 6         score ← exploit + λ · explore;
> 7         if max_score < score then
> 8             max_score ← score;
> 9             v_selected ← v_child;
> 10    return v_selected;   // 最有希望被探索的节点
> ```

算法 5 概述了 `Simulation` 的主要过程——返回奖励 r。基于 v_selected，该过程**迭代地扩展 MCTS 博弈树**直到达到模拟次数 S（第 3–6 行）。每次迭代中，被扩展节点对应的 CFG 表示 x 输入目标恶意软件检测系统 f，返回的奖励计算为 reward = 1 − f(x)（第 6 行）。

> **算法 5：Simulation(v_selected, f, S) 过程**
>
> ```
> 1  Procedure Simulation(v_selected, f, S)
> 2     v′ ← v_selected;
> 3     for i ← 1 to S do
> 4         v′ ← Expansion(v′);
> 5         x ← v′.x;
> 6         reward ← 1 − f(x);
> 7     return reward;
> ```

**(4) MalGuise 如何规避 ClamAV 的案例研究**。为揭示 MalGuise 如何规避真实环境中的反病毒产品，我们呈现一个案例研究——MalGuise 如何操纵一条野外恶意软件以规避 ClamAV 的检测。我们主要选 ClamAV，因为五种反病毒中四种是专有且闭源的，而 ClamAV 由 Cisco 开源 [16]。值得注意的是，对 ClamAV 而言，**一个可执行文件被报告为恶意软件**——当且仅当**它的任一签名与 ClamAV 病毒数据库（CVD）中的签名匹配**，CVD 持续以最新恶意软件样本更新。

为搭建案例研究，我们从 **WannaCry 勒索软件攻击** [1] 中随机选一条新出现的恶意软件变体，记作 WannaCry⁴。随后，如图 12(a) 所示，WannaCry 被提交到 ClamAV，并被**确定性地检测为恶意软件**——因为其 ".text" 节的哈希签名（即 "a5d0e70115ec99f7d770fe98861ca017"）**恰好匹配** CVD 中的签名。最后，如图 12(b) 所示，对 WannaCry 应用 MalGuise 的**一次 `call-based redividing` 变换**后，".text" 节的哈希签名变为 "939a8b36a29b27673de9ad4bd0623fed"——它**不再匹配 CVD 中的任何签名**，从而被**误识别为正常软件**。
