# Catch Me If You Can: Detector-Resistant Evasion via Semantics-Preserving Command Re-Realization

**有种你来抓：基于保语义命令再实现的抗检测器逃逸**

Muhammad Shoaib（University of Virginia）、Hare Sudhan Muthusamy（Independent Researcher）、Tareq Alkhatib（Fortinet）、Wajih Ul Hassan（University of Virginia）

---

## 摘要

红队需要逃逸技术来检验安全信息与事件管理（SIEM）检测规则的健壮性，但现有方案存在三方面缺陷：（1）依赖人工；（2）局限于字符串级混淆（如编码方案、引号技巧），很容易被反混淆器还原；（3）规则覆盖率有限。这使得"通过不同工具达成相同效果"这一类**保持语义的逃逸**长期未被探索，也就无法回答检测规则究竟在识别攻击意图，还是仅在匹配表层模式。我们提出 **SPECTRA**——一个自动化逃逸生成器，它在保持攻击效果不变的前提下，通过**功能等价的工具与参数结构**改写命令行实现。SPECTRA 不在语法层面打补丁，而是在语义表示上推理，从而自动产生更持久、更有效的逃逸。在 Windows Sigma `process_creation` 规则集上，SPECTRA 取得 72.9% 的规则覆盖率，而 AMIDES 仅为 37.6%（1.9×）；其生成的逃逸仅有 4.5% 能被反混淆器还原，AMIDES 则高达 78.1%（抗反混淆能力强 17.4 倍）。当评估对象切换到 SOTA 逃逸检测器、并取其零误报工作点时，SPECTRA 的被检率仅为 22.7%，而 AMIDES 为 69.9%。在所有指标上，SPECTRA 也优于五种通用 LLM。

## 1. 引言

红队的工作是模拟真实攻击，以检验组织的安全防御能否识别真正的威胁 [15, 26, 30]。此类评估的根本问题在于：SIEM 检测规则 [1, 9, 34] 究竟捕捉的是恶意行为的**意图**，还是仅匹配那些攻击者可轻易绕过的**表层模式**？SIEM 系统通过监控系统事件来工作，当观察到的活动满足规则在 `CommandLine`、`EventID` 等字段上的谓词条件时就告警 [7, 32, 36, 39]。Sigma 是用于在结构化事件记录上表达此类条件的主流规则格式 [32]。当攻击者完成了原本的恶意效果，但其行为已不再满足规则的检测条件时，就构成了对该规则的逃逸。理解检测规则在面对逃逸时的健壮性，是构建弹性防御的关键。

长期以来，安全研究者通过构造混淆变种来探究规则检测器的可绕过性 [46, 49, 66, 67]，方法包括各种编码方案（Base64、十六进制）、引号技巧、字符级伪装（CMD 中的脱字符 `^` 插入）以及包装器构造。其中代表性工作 AMIDES [66] 对 Sigma 规则 [1] 的逃逸技术进行了系统性研究——作者通过人工分析与手工编排逃逸变种，应用了 CMD、PowerShell 编码、引号技巧、脱字符插入等一系列字符串级转换，同时保留原命令的结构。

现有的人工逃逸方法在红队评估场景下暴露出关键局限。更一般地，要把**过程级逃逸**转化为一个**自动化系统问题**，必须解决三方面技术挑战。

**挑战 1：可扩展性与覆盖率有限。** 人工生成逃逸需要分析师逐条审查规则、理解语义、识别触发命令并手工构造变种。这天然限制了覆盖率：AMIDES 在 292 条 SIGMA 基准规则 [1] 中仅为 37.6% 的规则生成了逃逸，剩余 62.4% 完全未测试。规则集在持续演进，但人工方法跟不上这种节奏；对于动辄拥有数百乃至数千条规则的组织而言，全量评估在工程上不可行。

**挑战 2：逃逸脆弱，易被对抗手段还原。** 字符串级混淆几乎不堪一击：AMIDES 生成的逃逸中，78.1% 都能被现成工具反混淆 [6, 29]；在 SOTA 逃逸检测器 [66] 的零误报工作点上，69.9% 会被标记为可疑。这种脆弱性源自其手段过于浅薄——引号、脱字符、编码这些都是**句法层**的小动作，会留下与正常命令明显不同的特征模式；一旦归一化处理，原始的规则字面量就会重新浮现并触发告警。

**挑战 3：附带告警率过高。** 当把 AMIDES 的逃逸放到全规则集上评估时，会引发大量非预期告警——基准集上共触发 79 次附带告警，每个逃逸平均要触发若干次。这是因为字符串级转换会引入恰好命中其他无关规则的特征模式，从而降低其在真实攻击模拟中的实用价值。

综合来看，上述三点构成了本文要解决的核心技术问题。一个实用的逃逸系统必须能够：从一条触发规则的命令中**恢复其行为摘要**，识别那些**绝不能再出现的检测器字面量**，在不同工具与参数契约下**找到能实现相同行为的替代过程**，并把这些过程**实例化为 Windows 命令行**——且在整个过程中绝不让检测器关心的线索再次出现。

> **图 1：SPECTRA 的高层工作流**
> 输入两路语料：① 种子语料库（含规则 + 命令 + YAML）→ Rule Effect Extractor；② 参考语料库（来自文档化工具）→ Reference Processor。
> 两侧产出的"复合键 / 每函数工件"汇入 ③ Effect Matching Engine 进行匹配 → ④ 由 Evasion Synthesizer 合成 ⑤ 逃逸命令，最终输出给红队使用。

为应对上述挑战，我们提出 **SPECTRA**¹——首个完全自动化、**保持语义**的逃逸生成器。我们的核心洞见是：*健壮的安全测试需要语义等价，而不是句法混淆*——与其伪装攻击**怎么写**，不如检验规则能否识别攻击**做了什么**。先前方法在字符串上做手脚（如 Base64 编码），SPECTRA 则**在过程层转换**：替换工具、重塑参数、改造命令结构，同时保持攻击效果不变。这样才能真正暴露出检测规则到底是脆弱的模式匹配器，还是健壮的行为识别器。任务的高层抽象是：把一条**种子命令**（即最初触发规则的命令行）变换为一条新命令行，要求该新命令行**保留种子的效果**、**避开检测器匹配到的规则字面量**，并且**对替换后的工具来说仍然合法**。在工程实现上，SPECTRA 分别分析触发命令和已文档化工具的全集，通过共享的行为结构在两者之间做匹配，再把一个兼容的替代过程实例化为一条具体的逃逸命令。

> ¹ **SPECTRA** = **S**emantics **P**r**E**serving **C**ommand **T**ransformations for **R**ule **A**voidance（用于规避规则的保语义命令变换）。

SPECTRA 的输入为两类，输出为经验证的逃逸命令。在实验配置中：**种子语料库**包含 Windows Sigma `process_creation` 规则（即针对进程启动记录的单事件规则），并配对来自源日志的代表性触发命令；**参考语料库**则由文档化的系统工具组成，来源包括 LOLBAS [20]（一份记录了被攻击者改作他用的 Windows 二进制清单）、ss64 [35] 和 Microsoft 官方文档 [25]。如图 1 所示，SPECTRA 将这些输入交由四个组件依次处理：前两个组件**分别**分析规则侧与工具侧；第三个组件通过**结构化匹配**把两侧连接起来；第四个组件把一个兼容的匹配转化为一条具体的逃逸命令。图 3 在贯穿全文的"running example"上展示前两个组件的实例化结果，图 4 接续展示匹配与合成阶段。

**组件 1：Rule Effect Extractor（规则效果抽取器）。** 它的工作是从种子命令中抽取**语义效果**并识别一个**回避集**（avoid set）。给定一条规则及其触发命令，它推理的是命令"完成了什么"，而非"怎样写的"。一个 BiLSTM-CRF 标注器 [53] 将归一化后的命令切分为 FLAG / VALUE 跨段，赋以语义类型（path、url、dll），再由一个训练好的分类器把（flag-value-type）三元组映射到**原语操作**（read、encode、write）。例如 `certutil.exe -encode file.bin encoded.txt` 会被映射为效果序列 `read → base64_encode → write`。该处理器随后构建一个**效果图**——一种带类型的有向图，其边记录"哪些参数启用哪些原语"——并抽取**回避集**，记录该规则的字面量与匹配操作符（exact、contains、endswith、regex）。为避免相关性建模误差，SPECTRA 用两个不同初始化的抽取器实例并行运行，只保留两者在效果集与绑定链上**完全一致**的命令。

**组件 2：Reference Processor（参考侧处理器）。** 与组件 1 并行，它分析已文档化的工具集合，以发现"哪些已文档化的函数能够实现哪些效果"。对每个工具函数，它保留两个相互独立的视图：归一化的命令示例（**Example-View**）与周围的文档散文（**Anchor-View**）。它复用与 Rule Effect Extractor 中相同的**共识抽取器**对示例进行分析以构建效果图；同时由一个独立的文本编码器从散文中预测**参数模式**（schema）。这里的 schema 就是工具的**参数契约**：规定哪些 flag 必填、哪些可选；每个 flag 接受的值类型；flag 的元数；以及位置参数模板。一致性检查会将 Anchor 派生出的 schema 与 Example-View 的模式以及跨源一致性进行对照，仅保留通过这些检查的 schema。这样得到的 schema **植根于文档化的意图**，而不是任何单一模型的输出。

**组件 3：Effect Matching Engine（效果匹配引擎）。** 该组件将效果图映射到候选工具。其做法是用**复合效果键**对种子和参考函数同时建立索引；这些检索键编码了"包含哪些原语、各出现多少次、绑定链的粗结构"等信息。给定一个种子的效果图作为查询，该索引返回那些**能够以兼容的参数模式实现相同效果**的已文档化工具。索引强制三类约束：多重性一致（重复出现的原语必须数量匹配）、链形一致（绑定模式必须能对齐）、以及可用时的**跨二进制偏好**——以确保候选既保留行为，又不会退化为种子的平凡近似拷贝。

**组件 4：Evasion Synthesizer（逃逸合成器）。** 该组件挑选一个候选工具并合成具体参数。它把种子效果图与每个候选工具的**绑定摘要**（一个记录"该工具的哪些参数启用哪些原语"的紧凑结构）对齐，计算两个分数：**对齐分数**——衡量种子边中有多少能匹配到类型兼容的候选边；**覆盖分数**——衡量种子绑定链中有多少比例能找到兼容的实现。对齐为零或覆盖过低的候选直接被淘汰。在余下候选中，生成器按字典序应用如下偏好：效果图对齐 > 绑定链覆盖 > 精确键匹配 > 过程多样性（不同的二进制、不同的 flag 与位置模式）。一旦工具被选定，参数填充器在**回避规则字面量**的同时，依绑定摘要合成具体参数值。比如对于带有效果 `{net.fetch, fs.write}` 的 `certutil` 种子，生成器会选择 `curl.exe`，把 URL 与 PATH 分别对齐到相应原语，并填入 `-L -s` 与 `-o` 等必需命令行选项——实现完全相同的"下载+写入"行为，却不再触发原规则。

我们以 AMIDES 的人工逃逸为对照基线，在 292 条 Sigma `process_creation` 规则基准 [1] 上评估 SPECTRA。SPECTRA 为 72.9% 的目标规则生成了逃逸（AMIDES 37.6%）；被反混淆还原的概率比 AMIDES 低 17.4 倍（4.5% vs. 78.1%）；在完全规避规则的前提下，附带告警量降低 4.9 倍；完全规避的规则数是 AMIDES 的 1.92 倍（208 vs. 108）；面对 SOTA 逃逸检测器时的难度高 3.1 倍（22.7% vs. 69.9%）。我们进一步证明 SPECTRA 能够泛化到取自更新 Sigma 修订版的、新发布的 `process_creation` 规则上。此外，SPECTRA 在所有指标上也优于五个通用 LLM²。

> ² 引入 LLM 仅是为了把 SPECTRA 的表现放到"广泛可得的生成能力"这一上下文中比较，并不主张其相对于可能经过精调的模型为最优。专门的 prompt 策略或任务特定训练可能改善 LLM 结果；我们的目标是证明**专为该任务设计的语义推理**胜过人工方法与开箱即用的通用模型。

SPECTRA 的主要贡献如下：
- **自动化过程级逃逸**：首个针对 Windows Sigma `process_creation` 规则、生成保语义逃逸的系统。
- **效果图抽象**：一种命令表示形式，能够捕获原语及"参数 → 效果"的绑定关系，从而支持保效果的变换。
- **跨二进制合成**：带多样性偏好的生成机制，优先选取不同的工具、flag 和位置模式，而非停留在字符串级变体。
- **受 schema 约束的生成**：在学习到的 schema、绑定约束和回避集强制下进行自动合成与验证。
- **目标场景下的评估**：在 Windows Sigma `process_creation` 规则上的大规模评估，包括对更新 Sigma 修订版中新规则的泛化测试。

## 2. 预备知识

**核心记号。** 全文设计中始终使用四组小型记号家族。**检测器模型**侧：Φ 表示触发条件；R = (spec, Φ) 表示一条检测规则；`ĉ` 表示 SPECTRA 输出的逃逸命令。SPECTRA 的可复用核心则围绕一组反复出现的对象进行推理。**规则侧**：**回避集** `A_i` 记录"不能再次出现的检测器字面量"；**种子效果图** `G_i` 概括"需要保留的行为"。**参考侧**：**参考效果图** `G_u` 概括文档化的行为；**绑定摘要** `B_u` 记录"参数如何启用原语"；**参数 schema** `S_u` 记录工具允许的参数契约。匹配阶段后续会引入**复合效果键** K，合成阶段后续会引入兼容度度量 α、β。本文按上述顺序逐步引入这些对象。

评估设置为：通过一个固定的 PySigma/Azuma 后端渲染的 Windows Sigma `process_creation` 规则。当规则的触发条件 Φ 求值为真时即告警。我们用一个映射 E(c) 来建模命令的执行——它把命令 c 映射到一串有序日志事件，过程中应用 Windows 感知的归一化 N，包括：Unicode NFKC、可选的大小写折叠、flag 别名统一（`/f`、`-f`、`--flag`）、引号跨段保留、路径归一为反斜杠形式、以及对包装器的标注。在该模型下，一条逃逸命令**保留种子效果**，但让配对的规则求值为假，即 `Φ_R(E(ĉ))` = FALSE。除非另有说明，逃逸默认是**逐规则**的；当我们说**全局逃逸** E′ 时，指的是一个攻击者命令序列，满足对**所有规则** R′ 都有 `Φ_R(E')` = FALSE。

这种**事件局部**的规则匹配设置，与那些基于进程树、父子血缘或跨多事件构建的溯源图来推理的检测器有本质区别。要评估后者需要不同的基准，也需要不同的"保语义"定义——因为此时被保留的对象不再是一条单独的命令行实现，而是一个更大的多事件执行结构。

**命令陈述语言（CSL）。** 接下来我们固定整个抽取、匹配、填充、验证过程中所用的命令表示形式。CSL 用一段紧凑文法书写：`Statement ::= EXEC {ARG}`——其中 `EXEC` 记录启动器（已归一化的主二进制），`ARG` 是一个带类型的操作数多重集（包含 flag 与位置参数）。CSL 给每个操作数赋一个**种类**（kind），这一约定在全文反复复用；表 1 列出了这些种类及示例。CSL 是流水线中通用的表层表示——抽取器用它识别带类型的参数，匹配阶段使用其位置与绑定链结构，合成器与验证器则用它来实例化并检查一条具体命令行。在 running example 中，图 3 所示的 URL 与 PATH 参数就是 CSL 操作数，它们会在后续的种子与参考效果图中再次出现。

**表 1：CSL 操作数种类**

| Kind | 描述与简短示例 |
|---|---|
| FLAG | 不带值的开关；如 `flag="-O"` |
| PATH | 文件系统路径（文件 / 目录 / UNC）；如 `path="C:\Temp\a.exe"` |
| URL | 网络资源；如 `url="http://x/y"` |
| REG | 注册表键；如 `reg="HKCU\Soft\Run"` |
| NET | 显式套接字配置；如 `ip="10.0.0.5" port=53` |
| PAYLOAD | 编码 / 不透明二进制；如 `b64="SAFgA..."` |
| TASK | 调度器语义；如 `task="Backup" trigger="daily02(:)00"` |
| RAW | 某些二进制需要的偶发字面量；如 `txt="literal"` |

## 3. 设计

SPECTRA 由一个**可复用核心**加上若干**面向具体设置的适配器**构成。可复用核心处理效果图、schema 推断、效果匹配与受约束填充；评估所用的适配器则固定了 Windows 归一化规则、原语词表、进程创建事件 schema 与 Sigma 后端行为。设计顺序与图 1 的流水线一致。在高层视角下，整条流水线在回答四个问题：（1）种子诱导出什么样的行为与规则字面量；（2）已文档化的工具暴露出什么样的行为与参数契约；（3）哪些工具与种子兼容；（4）其中一个如何被实例化为具体的逃逸命令——分别对应 Rule Effect Extractor、Reference Processor、Effect Matching Engine 与 Evasion Synthesizer。

这四个组件之间，反复出现的核心对象就那么几个（沿用第 2 节的记号）。规则侧，流水线一路携带**回避集**与**种子效果图**；参考侧，它构建**参考效果图**、**绑定摘要**、**参数 schema**，以及一个仅用于排序的质量注释 `q_u`。随后匹配阶段用**复合效果键**与两项**兼容度度量**进行检索，合成阶段则把存活下来的候选排序，并把其中一个实例化为具体的逃逸命令。图 3 在 running example 上展示前两个组件的实例化结果，图 4 续展示查找、配对、排序、填充与验证。

在该设置下，SPECTRA 从两类输入起步。
**（1）种子语料库**：记 `S = {s_i}`，每个种子 `s_i = ⟨ r_i, c_i, y_i ⟩` 包含一个稳定的规则标识 `r_i`、一条代表性触发命令 `c_i`、以及对应的 Sigma YAML 块 `y_i`（参见图 3）。我们用公开 Windows 攻击语料（Mordor 及相关 Security Datasets [10, 11, 21]、Splunk attack_data [33]）构建 `S`，并借助 ATT&CK-to-events 映射 [28] 把带标签的进程创建事件与 Sigma 规则连接起来。
**（2）参考语料库**：`R` 是一组已文档化的系统工具，取自 Microsoft 文档 [25]、ss64 [35]、LOLBAS [20]；每个参考项 u 提供文档化的命令模板、参数描述与使用示例。

### 3.1. Rule Effect Extractor（规则效果抽取器）

Rule Effect Extractor 把每个种子转换成一份**结构化描述**，供下游使用。它从种子三元组（规则标识、示例命令、YAML）出发，先**归一化**命令、抽取**必须回避的规则字面量**，再把 YAML 与归一化命令对齐，恢复出一份"命令做了什么"的简洁行为视图。我们假设底层日志未被篡改；保障日志完整性不在本文范围内，部署时可依赖专门的系统 [63, 71] 来保护日志链路。基于该视图，抽取器输出本节其余部分都会用到的规则侧对象：构建一张捕捉"启用的动作与其参数绑定"的**种子效果图**，记录主二进制与**带类型的位置签名**，并派生出后续驱动检索的**复合效果键**。在该表示中，效果图捕获待保留的行为摘要；回避集捕获禁止再次出现的规则字面量；复合效果键则是传给匹配阶段的检索句柄。

#### 3.1.1. Seed Ingest & Context Linker（种子摄取与上下文链接）

我们把规则的**声明性上下文**与归一化后的种子命令绑定起来，让后续阶段能够按效果进行配对、并能强制 schema。本阶段任务很明确：在参考侧匹配开始之前，先把触发命令与规则 YAML 放进一个公共表示中。我们规范化 YAML `y_i`，并计算回避集 `A_i` = A(`y_i`)（带上操作符与大小写出处）。当 `y_i` 需要回填时，我们通过选用一个**与产生 `c_i` 的日志同期**的 Sigma 修订版（±Δ 天范围内；默认 ±14）来强制时序一致，并记录提交哈希与时间戳。然后依次执行 UNPACKSEED → NORM → SEEDLINK → NTOK → CMDEXT，并在 FAILCONS 处加门控。通过的种子产出**种子效果图**，连同主二进制和带类型的位置签名一起，供下游检索与对齐使用。

#### 3.1.2. Paired Extractors and Orthogonal Oracle（双抽取器与正交 oracle）

单个抽取器可能在种子与参考两侧犯**相关性错误**，进而削弱效果匹配与验证。为此我们训练两个抽取器实例：架构与标签指南完全相同，但**随机初始化**与训练样本的**随机置换**不同。每个抽取器返回 BIO 跨段、值类型、flag 元数，以及带绑定的效果集。仅当**两者**给出的效果集相同且绑定兼容时，我们才保留该命令——这沿用了先前工作 [43, 57, 72] 中基于一致性的可靠性信号。具体接受条件是：两组效果集完全相同，绑定链具有高重叠率；该重叠阈值在带 oracle 标签的分层开发子集上选择（详见 §4），以在保留下游配对覆盖率的前提下，稳定**一致性条件下的精度**。任何未能通过效果集一致性或绑定重叠阈值的条目，会在 FAILCONS（图 2 第 22 行）处弃权，不进入索引、不进入配对。为避免相关性错误，我们还在一个分层子集上用**不复用任何已学习抽取器信号**的正交 oracle 进行评估：该 oracle 检查 PowerShell AST 结构、schtasks XML 或物化参数、以及 reg.exe 的文法。我们汇报一致性条件下的精度，并给出 Wilson 95% 置信区间 [69]。实现细节与完整估计量见附录 A.5 与 §5.6。

#### 3.1.3. Seed Effect Graph Builder（种子效果图构建器）

当两个抽取器在种子的效果集与绑定上达成一致后，我们仍需一个结构化对象，把它带进检索、配对与验证阶段——为此我们为每个种子构造**效果图** `G_i`。形式上，`G_i` 是有向图，节点分两类：（i）§2 中的 CSL ARG 节点，附带其值类型，且在适用时附带 flag 元数；（ii）原语节点（属于 `P`）。当某参数依照 §4 所述的学习关联**启用**某原语时，我们从该参数节点向该原语节点添加一条有向边。规范键 k(`E_i`) 是原语清单 `E_i` 按一个固定全序去重并排序的结果。结合记录"参数→原语"边的**绑定链** `B_i`，效果图让我们能在后续检查候选：不仅保留效果集，还要保留**兼容的绑定模式**——这与先前工作中基于图的程序与恶意软件行为表示 [45, 56, 59] 一脉相承。

这些种子侧对象在流水线后续阶段各司其职：**排序后的原语清单**通过复合效果键参与检索；**效果图与绑定链**则被携带到后续做兼容性检查与语义验证。以图 3 中的 certutil 种子为例，其效果图 `G_i` 包含两条边 ARG(URL) → net.fetch 与 ARG(PATH) → fs.write；复合效果键的集合部分是排序后的原语清单 k(`E_i`) = `fs.write | net.fetch`，并附带链结构草图。仅当 `|E_i|` ≥ 1 且 `max_(p ∈ P) π_p` ≥ λ 时我们才为种子建立索引；否则——例如在 FLAG/VALUE 切分存在竞争、或由 `opaque_blob` 主导的情况下——系统会**弃权**。每条边可解释，因为它们都来自一个评分过的 (flag, value, type) 三元组。规范化与分词遵循 N；`M` 上的辅助后验作为对齐特征保留，但**不**纳入 k(·)。

#### 3.1.4. Text-based Calibration and Validation（基于文本的校准与验证）

我们**不执行** SPECTRA 生成的命令。取而代之，对每个函数 u = (bin, fn)，我们从两个文本视图监督：归一化后的示例调用 `E_u`（Example-View）与周围的散文 `A_u`（Anchor-View）。一个示例编码器和一个文本编码器分别基于这两个视图，预测出 `P` 与 `M` 上的后验。由于两视图至多**弱独立**，我们把它们之间的**一致性**作为协视图 / 多视图意义上的稳定性信号 [43, 57]，用以借助温度缩放（§4）[48] 校准阈值，并与验证器的 schema、类型、元数、规则字面量等检查共同作为接收门控。§5 给出最终的运行有效性。我们用三种方式强化该设置：**否定对照**——把 `A_u` 替换为某个 `u' ≠ u` 的 `A_(u')`，同时保持 `E_u` 不变；**释义测试**——对 `A_u` 与 `E_u` 做语义扰动；以及在存在多个公开来源时，检查**跨站一致性**。我们汇报一致性子集上的精度，附带 Wilson 95% 置信区间（CI）[69]，并在图 2 的 SANITY 中使用这些估计。任何未通过这些检查的候选，要么消耗 EVASIONSYN 中有限的重试次数之一，要么导致系统对该种子直接弃权。

### 3.2. Reference Processor（参考侧处理器）

**概述。** SPECTRA 的 *Reference Processor* 通过执行 RCRAWL → NTOK → REFGRAPH → BUILDIDX（图 2 第 5-8 行）来确定**已文档化工具如何实现相同的原语**。该阶段是 Rule Effect Extractor 的**参考侧对偶**：将文档转化为同样形式的对象，只不过对象是候选工具而非种子。在 running example 中，图 3 右半部分展示了这些对象。

#### 3.2.1. Reference Corpus (Documented System Utilities)（参考语料库——已文档化的系统工具）

我们通过 RCRAWL 与 NTOK 构建 `R = {r_u}`，按工具函数 u = (bin, fn) 建立索引。对每个函数保留两个文本视图：**Example-View** `E_u`——预格式化的命令调用示例；**Anchor-View** `A_u`——标题与周围散文。语料来源为 LOLBAS、SS64，以及 Microsoft Windows 与 PowerShell 文档 [20, 25, 35]。我们保留二进制名、函数标签，以及抽取得到的示例命令行的有序列表。所有示例都按 §2 中定义的同一映射 N 进行归一化；像 `<PATH>`、`<URL>`、`<DLL>` 这样的文档化占位符以 CSL 下的带类型 VALUES 形式保留。下游阶段消费的，仅是用与种子相同的共识抽取器从 `R` 派生出的工件；图 3 展示了 running 例子 `certutil → curl` 在该语料中的一个切片。

#### 3.2.2. Reference Crawler & Cache（参考爬取器与缓存）

我们通过爬取并缓存公开的 LOLBAS、SS64、Microsoft Windows 与 PowerShell 文档页面来构建 `R`。对每个工具函数 u = (bin, fn)，抽取命令示例与周围散文作为 `E_u` 与 `A_u`，并计算简单的、来自爬虫的统计量作为质量注释 `q_u`——后者用于配对与排序阶段。爬虫实现、缓存策略与视图构建细节见附录 A.2。

#### 3.2.3. Reference Effect Graph Builder（参考侧效果图构建器）

在效果侧，我们只使用 `E_u`。`E_u` 中每个归一化示例都通过**共识抽取器**（§3.1.2）产出示例级的效果集与绑定。函数层级上，**参考效果图** `G_u` 概括该工具的文档示例支持哪些原语；**绑定摘要** `B_u` 则压缩"这些示例中参数如何启用这些原语"。两者就是后续被匹配、配对、填充阶段消费的参考侧对象。函数级效果图 `G_u` 是示例边的并集（附支持计数）；仅由原型相似性支持的边被标注出来，以便置信度可以端到端传播。规范函数键 k(`E_u`) 按与种子相同的程序计算；保留绑定摘要 `B_u`；通过 MAKEKEY 形成复合效果键。后续配对至少要求种子图 `G_i` 与 `B_u` 之间存在一种**兼容绑定**，这比单纯的效果集相等更严格。对每个 (bin, fn) 我们都保留键、效果图、归一化示例和爬虫派生的质量注释 `q_u`。图 3 展示了 running example 的参考侧对象。

### 3.3. Effect Matching Engine（效果匹配引擎）

**概述。** 前面阶段在规则侧与参考侧产出了可对比的工件。Effect Matching Engine 把 MAKEKEY 生成的复合效果键**映射到工具函数及其 schema**，并由 LOOKUPIDX 查询（图 2 第 11 行）。它要回答的基本问题是：给定一条带特定效果集与绑定链结构的种子命令，哪些已文档化的工具能够以**兼容的参数模式**实现相同效果？

在本质上，本阶段是一个**基于共享中间对象的检索问题**，而非对命令字符串的直接改写。复合效果键将搜索范围收窄到具有相同原语清单与相同粗结构的过程；细粒度的参数兼容性留到配对阶段处理。因此匹配阶段遵循**由粗到细**的模式：先要求精确的键一致，再在配对阶段做参数级兼容性检查。用种子键查询索引，返回的是那些在 `P` 上与种子匹配、并且仍携带足够结构以便后续对 `S_u` 做参数验证的过程。在 running 下载例子中（图 4），效果 {`net.fetch`, `fs.write`} 的种子键检索回了 `curl`、`bitsadmin` 与 `PowerShell`。然后由多重性与链哈希一致性确保——只有效果计数匹配且粗绑定结构匹配的过程才会被传给对齐阶段。

#### 3.3.1. Composite effect key (MAKEKEY)（复合效果键）

复合效果键是索引使用的**粗检索句柄**，并非最终兼容性判定。我们写为 `K(G, E, π, B) = ⟨ k(E), κ(E), h_{{pos}}(π), h_{{chain}}(B) ⟩`。其中：k(E) 为排序原语清单；κ(E) ∈ {0,1,2}^`|P|` 记录原语多重性（上限为 2）；h_pos(π) 与 h_chain(`B`) 分别是位置签名与绑定链的 64 位哈希³。复合效果键的作用是**防止假匹配**：`⟨ k(E), κ(E) ⟩` 记录"有哪些行为以及各发生多少次"，`⟨ h_{{pos}}(π), h_{{chain}}(B) ⟩` 则记录"参数如何通过位置结构与绑定链来实现这些行为"。这四项合起来，让索引能在进入配对阶段前就排除许多显然不兼容的过程。我们要求**所有键成分都精确匹配**：原语清单、多重性、位置哈希、绑定哈希都必须一致。这种保守策略确保**进入下一阶段的候选已经在效果身份与结构上与种子匹配**。

> ³ FNV-1a 哈希，固定种子；实验中未观察到碰撞。

#### 3.3.2. Argument schema per (bin, fn)（每个 (bin, fn) 的参数 schema）

对每个工具函数 u = (bin, fn)，我们派生出一个**参数 schema** `S_u`，它在 CSL 下总结被接受的 flag、必填性、可接受的值类型、元数与位置布局。我们从 Anchor-View `A_u` 估计它——这样**效果抽取与 schema 推断使用的是不相交的视图**：示例说"工具做什么"，散文说"应该怎么调用"。文本编码器预测候选的 flag 集合、可接受的值类型、位置签名与 flag 元数；一致性阶段再把这些预测对照 `E_u` 与（可用时）跨源一致性进行检查。最终的 schema 把基于 Anchor 的预测与由 `E_u` 派生的绑定摘要 `B_u` 结合起来，作为合成器必须满足的**契约**。

**Schema 一致性。** 每个由 Anchor 派生的 schema 都要经过一致性阶段——把它与归一化示例 `E_u` 比较，可用时还与跨源一致性比较。该阶段会**调整或拒绝**那些与强证据冲突的 flag、元数、值类型与位置模板，仅当所有检查都通过时才把 schema 标记为一致。精确的估计量、阈值与编辑规则见附录 A.7。

**索引内容与查询。** 每个索引条目存储工具标识，并附带其 schema `S_u`、质量注释 `q_u`，以及供配对与后续分析使用的元数据。我们用种子的复合效果键查询索引。本文所有结果所用的配置中，LOOKUPIDX 仅返回**复合效果键与种子键完全匹配**的工具——也就是说集合键 k(E)、多重性草图 κ(E)、位置哈希 h_pos(π)、绑定链哈希 h_chain(`B`) **四者全须一致**。若找不到这样的桶，查询返回空候选集，EVASIONPIPELINE 对该种子弃权（图 2 第 11-12 行）。

**共识门控。** 仅当 Ext₁ 与 Ext₂ 在效果集上一致、且绑定链重叠达到至少 τ_bind（§3.1.2）时，我们才把一个种子或函数条目纳入 `I`；任一检查未通过的条目都不被索引。出于审计目的，每个条目都记录一致性位与绑定重叠得分。在分层子集 `S^*` 上，我们还运行正交 oracle ORACLE（PowerShell AST、schtasks XML、reg.exe 文法），要求**共识赋值满足对应函数特定的约束**。这正是 §3.1.2 中定义**一致性条件下精度（ACP）**所用的相同设置，§5 给出最终的 ACP 数值。

### 3.4. Evasion Synthesizer（逃逸合成器）

Effect Matching Engine 给出效果兼容的若干工具后，Evasion Synthesizer 把其中**一个**兼容过程转化为具体的逃逸命令。分三步：**配对、选择、填充与验证**。图 4 从左到右依次为查找、兼容性测试、排序、参数填充与验证。种子效果图与回避集固定"必须保留 / 必须排除"的内容，候选工具的绑定摘要与 schema 则规定该工具如何合法地实现这一行为。

利用种子侧对象与精确键查找返回的候选集合，生成器选定工具、合成参数并验证最终的命令行。图 4 展示了下载例子中的 J(u) 与参数填充器。每个输出都以 `⟨ c_i , ĉ ⟩` 对的形式记录，便于下游分析"字面量复用程度"与"新颖性"。

**输入与契约。** 根据 Effect Matching Engine 的构造，每个候选 u 已经在效果集上与种子匹配。生成器消费候选 schema `S_u` 与绑定摘要 `B_u`，可发出一条或多条 Windows 命令行，要求：首个 token 为候选二进制；满足 schema；遵守种子回避集 `A_i` 与种子图 `G_i`；通过 §4 中的验证器门控；并保留基于抽取器的语义。也就是说，**生成是由候选 schema、绑定摘要与回避集驱动的，而不是对目标行为本身做修改**。在 running example 中，填充器以种子的 URL 与 PATH 实例化 `S`_{u*}，得到 `curl.exe -L -s [URL] -o [PATH]`，同时回避 `A_i` 中的所有字面量。形式上，每条逃逸命令 `ĉ` 都须对配对规则 r 满足 `Φ_r(E(ĉ))` = FALSE；生成阶段对其他 R′ ≠ r 上的 `Φ_(R')(E(ĉ))` 不作要求，跨规则结果在 §5 汇报。

#### 3.4.1. Candidate Pairing（候选配对）

在 EVASIONSYN（图 2 第 31-45 行）内部，被检索到的候选已在**效果**上匹配，但参数层面仍可能不兼容。为此我们把种子效果图 `G_i` 与每个函数的绑定摘要 `B_u` 对齐，丢弃那些参数无法实现种子绑定的工具。配对要回答的问题是：**候选是否暴露出与种子相同的"原语→参数"结构所需的参数槽位**？我们计算**效果图对齐分数** α(`G_i , B_u`) ∈ [0, 1]——通过求解**边上的最大二部图匹配**得到。一侧为种子边、另一侧为候选边；只有当原语匹配且参数端点 schema 兼容时，一对边才有资格匹配——位置参数要在类型与相对顺序上统一，flag 必须接受所观察到的值类型与元数。一个学习到的打分器给可匹配对加权，低兼容性的对被忽略，α 即"被最大匹配覆盖的种子边的比例"。对齐分数为零的候选被丢弃——即使效果集相同，它们也无法以兼容的方式实现种子的绑定。

我们另外测量**绑定覆盖率** β(`G_i , B_u`) ∈ [0, 1]——在相同匹配下，`B_u` 中能为种子绑定链提供**类型兼容**实现的比例。如图 4 所示，α 与 β 把"仅原语清单相同"的候选与"参数结构能够真正实现种子行为"的候选区分开来。低于固定对齐 / 覆盖率阈值的候选被丢弃。匹配细节与阈值见附录 A.6 与 §4。

#### 3.4.2. Diversity Bias（多样性偏好）

为鼓励**结构性变化**，EVASIONSYN 仅在通过配对建立**语义兼容性之后**再施加多样性守卫。对每个候选函数，它测量三件事：二进制是否变化、flag 集合相对于种子有多不同、位置模式相对于种子有多不同。**只要存在至少一个跨二进制选项，就只保留跨二进制的候选**。在所有情况下，候选要被传给选择器还须达到足够的 flag 新颖性、位置新颖性、对齐分数与绑定覆盖率。精确统计量与阈值见附录 A.6，实验所用固定值见 §4。表 5 的配置 H 把这些守卫作为一组进行消融，结果显示——禁用它们只换来很小的覆盖率提升，却显著加重了脆弱性与附带告警，因此默认配置中保留它们。

**字典序选择。** 对齐与多样性过滤后，我们得到一个小的可接收工具子集。生成器按**字典序**对这些候选排序：先最大化效果图对齐，再最大化绑定链覆盖率，然后查阅图 4 中显示的固定检索层字段，最后才偏好更高质量、更具多样性的候选。该评分向量记作 J(u)。在本文采用的精确复合键查找配置下，所有存活候选的检索层字段为零，因此不影响排序。换言之，**排序是在一个语义可接收的集合中做精修，而不是去修复不兼容的候选**。这种"约束优先"策略避免了引入学习到的混合权重，并保证选择**可复现**。

#### 3.4.3. Argument Filler（参数填充器）

一旦 u* 被选定，FILLARGS 就用种子绑定链 `B_i` 与回避集 `A_i` 实例化 `S`_{u*}，并可被多次调用，为同一对 (种子, 工具) 生成不同变种。输出为参数表层串 a_{u*}——把它接在 u* 的二进制之后即得到候选命令行 `ĉ`。我们保留原始种子 `c_i`，以便验证阶段能比较 `⟨ c_i , ĉ ⟩`。实际上，我们处理的操作数种类很小：flag、path、URL、registry、网络端点、payload、调度项、字面量（表 1）。记 `L ( S`_{u*}) 为 schema `S`_{u*} 允许的参数串集合。填充器构造一个参数序列，**遵守 `S`_{u*} 的必填与可选部分，并遵循位置模板 π**。一个候选参数串被认为可行，当且仅当：每个槽都能填入 schema 允许的值，且按已记录的操作符语义（exact、contains、endswith、regex）与大小写策略，其 token **不命中 `A_i` 的任何元素**。

**仅有可行性还不够。** 最终命令必须在共识抽取器下诱导出**相同的效果集**，并在容差范围内仍与种子绑定链对齐。实例化进一步被候选绑定摘要 B_{u*} 偏置——当存在多个 schema 合法的实现时，B_{u*} 帮助在可选 flag 与位置参数之间做出选择。一个生成式组件提出与 E_{u*} 中规范化示例类似的参数串，同时尊重种子侧绑定与回避集。我们用 ASMCMD 拼装最终命令行，用 SANITY 检查，并允许一个小的重试预算 k（图 2 第 39-45 行）；否则系统对该种子弃权。

## 4. 实现

**抽取器与训练。** 抽取器的实现是为 Windows 命令行打 BIO 跨段，标记 FLAG 与 VALUE，并按二进制族强制做**互斥切分**——这样从种子和参考派生出的样本会落入分离的折（fold）中。抽取器输出：每个 VALUE 的值类型分布 τ、每个 FLAG 的元数标签 `â_f`，以及一个 (flag, value, type) → `P` 中原语的映射（§3.1.2）。两个不同初始化的实例（Ext₁、Ext₂）并行运行，下游阶段只消费两者效果集一致、绑定链重叠程度高的样本——重叠按"两组绑定链集合之间的 Jaccard 相似度"度量，超过一个固定阈值即视为高重叠；该阈值通过对一个分层开发子集进行小规模网格搜索（以正交 oracle 标签为参照）选定；选取**一致性条件下精度（ACP）已经平稳**的最小阈值，同时保留足够的样本供下游配对（实验中为 0.8）。flag 元数则采用**层级回退估计**——跨二进制、跨函数、跨 flag 上下文做插值，并由当前命令的贡献加 Laplace 平滑；未见 flag 回退到由参考统计派生的先验，验证器最终强制使用所推断的 `â_f`。表 5 的配置 L 禁用了负对照与弃权机制，结果显示覆盖率提升但置信度与反混淆率均变差——与"保守的共识门控应当保留"这一立场相一致。

**索引与检索。** 我们构建一个由复合效果键 K(G, E, π, `B`) 索引的效果匹配引擎（来自 §3.3）——它在纯集合键之上扩展了多重性、位置与绑定链草图。每个条目还存储函数 schema `S_u` 与在基于文本的监督协议下推断出的绑定摘要 `B_u`。检索通过纯哈希查找：LOOKUPIDX 仅返回复合效果键与种子键**完全相同**的工具；若桶为空，则查询返回空候选集，流水线对该种子弃权。候选只有在 §3.1.2 的共识门控成立时才被传递到配对阶段。

**选择、填充与阈值。** 从检索集合中，我们先按 §3.4.2 描述的多样性守卫形成一个可接收子集，然后按 §3.4.1 那样计算效果图对齐分数 α(`G_i , B_u`) 与绑定链覆盖分数 β(`G_i , B_u`)。我们在一个验证子集上选定**边兼容性剪枝阈值** ε 以及 (α, β) 的可接收水平，然后固定下来供所有实验使用。我们随后按字典序按 (α, β) 选定函数 u*，合成一个满足 `S_u` 的参数串 `a_u`——它要尊重种子侧绑定与回避集 `A_i`——并把生成的命令行交给验证器，由它强制 schema 一致性、值类型、元数与回避集检查。§3.4.2 中的多样性谓词 ADM 采用固定阈值 (τf, τπ) = (0.4, 0.3) 分别约束 flag 与位置的新颖性——这是在验证子集上通过一次小规模网格搜索选定并随即冻结的。表 5 中的配置 H 与 M 禁用多样性守卫或移除剪枝，结果是覆盖率提高，但附带告警增加，完全规避规则的比例下降——这支持把这些固定设置视为**保守的默认值**，而不是每个工作负载都需要去调的旋钮。

**校准细节。** 我们在一个验证子集上，通过最大化精度加权的 Fβ 分数（β < 1）来选定原语决策阈值 λ 与"参数→原语"评分头中的混合权重；该验证子集的筛选条件是：两个编码器都把 p ∈ `P` 预测得高于各自的决策阈值。这些设定在所有实验中保持不变。然后我们在同一个由**文档级一致性**定义的验证子集上，用温度缩放对两个编码器头做校准，并把校准后的得分用于后续所有消费编码器输出的组件。校准只调整两个抽取器头的决策温度；共识、对齐与剪枝阈值一旦选定后保持不变。表 5 的配置 C 禁用了温度缩放——结果呈现出"覆盖率略高、置信度略低"的预期模式，符合"校准只是单一全局调整，而非在评估集上调过的旋钮"这一定位。

## 5. 评估

我们沿九个研究问题评估 SPECTRA：
（1）逃逸生成覆盖率；
（2）生成的逃逸对反混淆工具的鲁棒性；
（3）目标规则与附带告警率；
（4）Windows 上的运行有效性；
（5）整体检测性能；
（6）生成逃逸的语义等价性；
（7）组件消融；
（8）对新发布 Sigma `process_creation` 规则的泛化能力；
（9）与基于 LLM 基线的比较。

这些问题各有目的，按上述顺序呈现。我们先问系统是否能产出**看似合理**的 Windows 命令行；再测试它在归一化、跨规则告警与学习到的逃逸检测器面前的鲁棒性；接着检查生成的命令在执行上是否仍可运行且语义等价；最后给出消融与时间泛化的结论。

我们采用与 AMIDES 研究 [1] 一同发布的 292 条 Sigma 规则，它为 Windows `process_creation` 规则提供逃逸。AMIDES 在这套固定集合上评估并汇报结果；据我们所知，它是目前唯一**端到端、面向 SIEM 规则、直接生成可规避的 Windows 命令行**的系统，因此我们把它当作该任务的 SOTA。还有更广泛的一类工作研究命令行混淆——既不消耗 SIEM 规则、也不针对固定规则集给出覆盖率估计的——包括参数混淆器和脚本级转换器 [2, 16, 18]；AMIDES 已在不同混淆风格下评估了规则鲁棒性。然而这些框架既不以 SIEM 规则为输入，也不设计为在一个固定规则集上提供覆盖估计，因此**不构成本评估的直接基线**。本基准考察的是 Windows Sigma `process_creation` 规则的**族内时序泛化**，不是跨类别或跨语言泛化。

### 5.1. 逃逸生成覆盖率

我们以**覆盖率与一致性**开篇——因为后续指标只有对那些"接收到看似合理 Windows 命令行"的规则才有意义。**生成覆盖率**指 292 条规则中，系统至少能生成一条候选的比例（表 2 的 Gen.）。**Windows 一致性率**是指所有返回候选中，通过 Windows 单行一致性检查的比例（按表 2 中 Valid /（Valid + Invalid）计算）。一致性检查包含三项确定性检查：单行、PowerShell 与 Windows 命令 shell 下的引号 / 转义平衡 [22, 23, 24]、以及 Windows 目标上下文。这些检查**故意做得最小**，主要是因为通用 LLM 经常输出畸形或非 Windows 字符串。由于我们在 §B.1 也对 LLM 应用同样的覆盖率与一致性度量，所以这里对 SPECTRA 与 AMIDES 应用相同检查以保持公平。这些检查**不能**替代执行或语义验证——它们只是防止幻觉输出**虚抬**后续实验数据的护栏。最小正确性在此处已经足够，因为后续 §5.4 通过实际执行衡量 Windows 上的运行有效性；§5.1 的目标只是为后续实验提供"能被公允测试的候选"。表 2 显示 SPECTRA 至少为 72.9% 的规则产出一个逃逸，AMIDES 为 37.6%。在各系统生成的候选中，AMIDES 未通过 Windows 单行一致性检查的比例为 6.4%，SPECTRA 为 2.8%。这一选择意味着后续的改进**可归因于语义变化，而非格式手段**。

### 5.2. 对反混淆的鲁棒性

接下来我们问，生成的逃逸能否在通用反混淆工具 [6, 29, 38] 的作用下幸存——抑或被还原为可识别的规则触发。我们定义**脆弱性**为"被反混淆工具还原、或塌缩回原意"的候选比例；该比例越高，意味着逃逸越容易回到原始意图，对真实防御几乎不构成保护。AMIDES 候选在 78.1%（400/512）的情况下被反混淆（见表 2），而 SPECTRA 候选仅有 4.5%（36/804）。换句话说，如果一个候选在通用归一化下塌缩，则防御方实际看到的就是原始意图，逃逸便丧失了价值。SPECTRA 跨二进制、跨函数地改变**过程与参数结构**，而不靠字符串层的小手段——因此对这些工具基本无可还原之处，从而呈现出**远低**的脆弱率。

**图 5：AMIDES 逃逸命令的反混淆分类。** 类别 A = 纯 CMD 技巧；B = 仅路径归一化；C = PowerShell 重度编码；D = CMD 与 PowerShell 混合；E = 已经可读（无混淆）。

**AMIDES 为何容易被反混淆？** 图 5 给出按类别的还原分析——能解释 78.1% 的反混淆结果：纯 CMD 技巧是最大类别（198/202 被引号、脱字符与环境变量归一化还原）；PowerShell 重度编码贡献 169/186 的还原（这正是常见解码器解开 `-enc` / base64 载荷与字符串运算符滥用的场景）；路径归一化数量较小但大多可还原；混合 CMD 与 PowerShell 包装器多为部分撤销；而已经"合法可读"的那类自然没有可反混淆之处。简言之，AMIDES 倚赖字符串与编码层的变换，而这些恰恰是上述开源归一化器**设计用来反转**的。

### 5.3. 全规则集附带告警

接下来我们追问——压制目标规则是否仅仅把告警转移到了规则集中的其它规则上？对每个候选我们都模拟 Sysmon 事件并对所有 292 条规则求值。我们计算：**全规则集告警总数** Htot、**目标规则绕过率** RTB (%)（即被种子触发但被逃逸所规避的目标规则比例）、**完全规避规则数** RCB（即至少存在一个候选不会触发该规则集中任何规则的规则数）；以及**每个完全规避规则的平均告警数** Havg = Htot/RCB。我们在表 2 中同时汇报 RCB 的**数量**与**占 292 条规则的百分比**。所有告警都用 Azuma 计算——这是一个基于 PySigma 的匹配工具，在固定的规范化下渲染 Sigma 规则。

两套系统都达到 100% 的目标规则绕过率。差距出现在**附带告警**上：SPECTRA 把告警总数从 79 降到 31（少 60.8%），并把每条完全规避规则的告警从 0.7315 降到 0.1490。完全规避规则的数量从 108（37.0%）上升到 208（71.2%）。这表明 SPECTRA 不仅击败了目标检测器，还**回避了由其它规则触发的溢出告警**——这是比表 2 中规则覆盖率更强的概念。

一个自然的担忧是：AMIDES 在同一套 292 条规则上使用了手工编排的技巧，可能在该数据集上过拟合。与 AMIDES 不同，**SPECTRA 是完全自动化的**：它训练一个共享抽取器一次、从已缓存文档构建效果图与 schema、按复合效果键配对、并在固定验证器下填充参数——没有任何**逐规则**的调参或人工编辑。固定的模型参数与检测阈值限制了规则特定的泄露，使得**评估协议在不同次运行间保持一致**。

### 5.4. 运行有效性

本小节追问：通过此前字符串级检查的候选是否还能在自动化测试台上作为合法的 Windows 命令行**真正执行**。

**错误分类。** 我们把"需要改写调用方式才能修复的语法 / 用法错误"与"由环境依赖导致的故障"区分开。一个命令仅当进程退出码属于 Windows 与相关工具族**为非法绕行 / 畸形命令行预留的代码**：2、87、1639、−2147024809（`0x80070057`）、或 −196608（`0xFFFD0000`），才算 **Not valid**。对 MSI，我们额外仅在错误文本与已知用法 / 解析模式匹配时才将 1619、1620 视为语法相关，否则视为环境相关。其余所有失败——包括缺文件、拒绝访问、离线资源、缺二进制等——都被视为**Inconclusive 不计入分母**。我们安装所需工具，并以**非交互方式**回放命令。仅当命令执行时未触发上述任一语法 / 用法码、未触发等价解析错误时才被记为 **Valid**；仅当确实出现该类错误时才记 **Not valid**。那些只能通过改变运行时环境才能解决的故障，从不被罚分为非法命令行。

### 5.5. 逃逸检测性能

本小节追问：一个**为恢复逃逸而专门设计的学习检测器**是否仍能识别 SPECTRA 的变体。AMIDES 提供了一个 SOTA 检测器——它从规则与正常样本中学习，恢复仍接近规则签名的变体 [66]。我们对比的其它系统都不包含逃逸检测器，因此沿用 AMIDES 发布的检测器来度量"各系统的逃逸有多少仍被标记"。图 6 显示 AMIDES 与 SPECTRA 在我们扫描决策阈值时的精度、召回、F1、MCC 曲线。在 AMIDES 报告的零误报工作点上，该检测器标记 69.9% 的 AMIDES 逃逸，但仅标记 22.7% 的 SPECTRA 逃逸。AMIDES 的曲线一致地更高——这与检测器被调适于 AMIDES 一致；而对 SPECTRA，一旦把阈值从极低水平上提，召回会迅速下降。AMIDES 的检测器使用向量嵌入做相似度评分，往往在"与规则或训练样例存在大量词汇重叠（如长共享子串或路径）"时被触发。SPECTRA 在保持底层效果不变的同时，**跨二进制与函数地改变了过程与参数结构**——词汇重叠大幅降低，因此一个窄的"规则相似度检测器"很少命中。这一模式与 §5.1 一致：通用字符串归一化器还原了 78.1% 的 AMIDES 案例，但仅还原 4.5% 的 SPECTRA 逃逸。

**图 6：AMIDES（SOTA 自适应逃逸检测）对各系统逃逸的检测结果。** D.R. = 检测率，按 AMIDES 协议在无 FP 的点上计算。(a) AMIDES 检测率 69.9%；(b) SPECTRA 检测率 22.7%。

### 5.8. 对新发布 Sigma `process_creation` 规则的泛化

我们以一个问题收尾：**同一套冻结的流水线是否能迁移到来自同一检测器族的更新 Sigma `process_creation` 规则上**。本实验有意做得很窄：它检验同一流水线是否能在**无需重新调参**的情况下迁移到新发布的 Sigma `process_creation` 规则。我们从最新 Sigma 提交 [19] 中随机采样十条 `process_creation` 规则，把它们当作**未见种子**处理，并以主评估中所用的同一阈值与验证器运行完整的 SPECTRA 流水线。SPECTRA 为其中 8/10 的规则生成了逃逸；所有 8 条都通过 §5.1 的一致性检查，没有一条被 §5.2 下的反混淆标记，且全部 8 条都绕过其目标规则**且不引入额外附带告警**；§5.3 中 8 条中的 7 条还通过了 §5.4 的运行有效性检查；7 条中没有一条被 §5.5 的 AMIDES 逃逸检测器检出。两次失败遵循与主评估相同的语义驱动失败模式：一条是关于"可疑部分编码调用"的 PowerShell 规则（rule id `536e2947-3729-478c-9903-745aaffe60d2`）——共识抽取器因混淆后的载荷无法产出稳定的效果图而弃权；一条是关于 `MpCmdRun.exe` 的 Windows Defender 定义移除规则——附带 `-RemoveDefinitions -All`（rule id `9719a8aa-401c-41af-8108-ced7ec9d75c`），其回避集与已文档化的 schema 不留下任何"效果兼容"的替代工具。这些结果只支持**族内泛化**的论断。

## 6. 相关工作

**规则逃逸检测。** AMIDES [66] 提供了 SOTA 规则逃逸检测器——它把事件比对 SIEM 规则与正常流量，以揭示**接近误匹配**的旁路。其基于学习的相似度评分器在"接近误匹配的旁路"上表现最佳；SPECTRA 的**跨二进制、保语义**变换则在保留效果的同时远远偏离了那类表层模式。我们的评估显示，SPECTRA 的逃逸**难以被检出**，而 AMIDES 自身的逃逸则常被抓到。网络 IDS 逃逸研究 [40] 针对的是**对抗性流量**，而不是进程创建语义。另一些实务派技术演示了通过重命名、脚本编辑、shellcode 变形 [13, 37, 65] 来做的签名旁路，但它们针对的是不同输入（网络特征、字节模式），且并不**自动生成保效果的逃逸**。

**LOLBin 检测器。** LOLBins 检测器 [44, 62] 对命令做分词，并在词法特征与 n-gram 上训练分类器。它们是**检测器，而非生成器**；其使用基于词表 / 词频特征——既不建模跨工具等价性，也不强制 schema 合法性。在我们的威胁模型下（语义固定、实现可变），名字 / 正则驱动的检测会失败——因为保效果变换会**替换工具、重塑参数**，跨越了词法决策边界。这类系统需要预定义的类别和分析师投入 [66]；SPECTRA 则**完全自动**运行。

**基于溯源的检测与调查。** 在检测侧，基于溯源、基于图的系统 [41, 51, 52, 58, 60, 70] 在进程树、血缘特征、上下文相关性、以及溯源 / 因果事件图上推理，而不是单条命令行实现。它们因此与我们的设定相关，**但不是本基准的直接基线**——本基准围绕事件局部 SIEM 规则匹配（如 `CommandLine`、`Image`、`ParentImage`）构建。在调查侧，基于溯源的调查系统 [42, 50, 54, 55, 64, 68] 重建多事件执行历史——同样**正交**，因为它们支持**事后分析**（且建立在更丰富的事件结构之上），而不是单事件规则匹配。我们的结果因此不对这两类系统的有效性做评判。

**反混淆与规范化。** Invoke-DOSfuscation [17]、Revoke-Obfuscation [31]、CMD-DeObfuscator [6] 与 PowerDecode [29] 这些工具会**塌掉表层编码、引号与别名**，使行为保持不变。SPECTRA 则**改变过程本身**（不同工具与带类型的参数）**同时保留效果**。因此**字符串级反混淆与我们的逃逸正交**：即便经过归一化，被实现的过程也不同，那些绑定到具体二进制 / flag 的规则谓词便不再触发。

**Pyramid of Pain。** Bianco 的"Pyramid of Pain" [3] 把工具 / TTP 放在**最难对付的防御信号**的顶端，原子指标（哈希、IP、字符串）位于下方。SPECTRA 正是面向这一顶层——通过在保持行为不变的同时**重写过程**（替换工具与参数）。先前工作 [2, 16, 18] 与 AMIDES [66] 针对的是**低层**（重命名与正则微调即足以应付的层）。我们的**过程级、保效果**的变换则瞄准防御方最优先关心的"难变更的工具 / TTP 层"——而我们的评估显示，这恰恰是 SOTA 防御**失误最多**的地方。

## 7. 局限与未来工作

**评估设置的范围。** 我们的结论限于**规则化 SIEM 检测**——也就是匹配事件局部字段（如 `CommandLine`、`Image`、`ParentImage`、`EventID`）的规则。我们**不**评估那些推理多事件结构的检测器（如进程树、父子血缘链、跨多事件构建的溯源 / 因果图）。本文的发现应被解读为关于"以命令行为中心的规则匹配"的证据，而**不**是关于所有检测器族的论断。

**防御策略。** 我们的发现凸显了字符串型规则的极限，并指向"建模行为而非表层"的防御方向。有前景的方向包括：基于行为、植根于因果关系的检测 [52, 60]；能识别"跨用户、跨血缘、跨时间"中等价过程的**效果级签名** [70]；以及在效果图等语义表示上结合规则 + 学习的混合体 [41]。设计与评估此类防御超出本工作范围，但 SPECTRA 可以充当一个**测试床**，用来度量它们面对**程序上多样化攻击**时的健壮性。

**对其它 Sigma 规则类别的扩展。** SPECTRA 当前聚焦于 Sigma 的 `process_creation` 事件。扩展到 Sigma 的文件、注册表、网络规则只需**修改适配器**，无需改动核心检索 / 合成流水线。具体而言，原语词表、归一化规则、事件 schema 与参考语料需要扩展，以覆盖文件创建 / 修改、注册表写入 / 删除、以及 DNS 查询 / HTTP 投递等网络动作。同一套效果图、schema 推断、效果匹配、受约束填充机制将随之在这些新适配器上工作。

**对其它规则语言与 Linux 命令行的扩展。** 支持其它规则语言（如 Elastic 或 Splunk），或从 Windows 迁移到 Linux 命令行，同样只需要在**渲染器与后端边界**、字段映射层、归一化阶段、原语词表与参考语料上做适配工作。对 Linux 而言，归一化层尤其需要建模 shell 特定语法（管道、重定向、命令替换）；参考侧则可取自 `man` 页与 GTFOBins。我们把这些适配实现及其评估留待未来工作，**不**主张它们已在本文得到验证。

**BiLSTM-CRF 对比 Transformer。** 如附录 A 所述，跨段检测使用 BiLSTM-CRF 标注器做命名实体识别 [53]。我们的输入是短 Windows 命令行（几到几十个 token），所以双向 LSTM 足以捕捉相关局部上下文，CRF 层强制跨段边界一致。我们在相同标注语料和留出验证切分上训练了一个**等容量的两层 Transformer 编码器**（附录 A.3）。Transformer 在跨段 F1 或下游一致性条件下精度（我们的设置中 ACP = 95.2%）上**没有带来提升**，反而带来更高的训练与推理代价。因此我们采用更简单、更稳定的 BiLSTM-CRF 架构。

**种子语料与参考语料的可用性。** 两套语料都**基于公开来源**构建。种子语料合并了公开基准所用的同一 292 条 Windows `process_creation` Sigma 规则，与来自 Mordor、Splunk 及相关开源攻击轨迹 [10, 11, 21, 33] 的代表性命令；要复现只需这些数据集与 §5 中固定的 Sigma 提交。参考语料则爬自 LOLBAS、SS64、Microsoft Windows 与 PowerShell 文档 [20, 25, 35]——对每个已文档化的二进制与函数，我们收集示例命令行与周围散文，然后规范化并按效果图建索引；该语料对**已文档化的 Windows 命令行行为**而言是**公开可复现且覆盖广泛**的。

**范围：为什么不做包装器级逃逸？** 我们的设计把生成**限定在单工具过程**，不合成 shell 包装器或控制操作符（如 `cmd /c`、`powershell -Command`、`&&`、`||` 或管道）。这让基于抽取器的等价性**与 shell 特定的解析与规范化解耦**，把搜索空间聚焦于**工具级变化**。基于包装器的逃逸因此与本工作**正交**，留待未来工作。

## 8. 结论

SPECTRA 把"SIEM 逃逸"重新表述为**自动化的、保语义的命令转换**，而不再是人工的规则逆向工程。通过**按效果而非按表层模式**配对过程，该系统在评估设置下取得了比手工基线**更广的覆盖率、更强的弹性、以及更好的泛化能力**。我们的工作确立：**文档驱动的合成可以系统化地、大规模地生成运行有效的逃逸**。
